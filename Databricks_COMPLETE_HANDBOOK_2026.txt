A1. Why does distributed computing exist?

	A single machine has limited CPU/RAM/disk. Once a dataset is too big to process on one machine
	in reasonable time (or doesn't fit in memory), you split the data across many machines and
	process the pieces in parallel. That's "big data" processing in one sentence.
	Hadoop MapReduce was the first mainstream version of this. It wrote intermediate results to
	disk between every step, which is safe but slow. Apache Spark was built to keep intermediate
	results in memory across steps, which is why it's typically 10-100x faster than MapReduce for
	multi-step jobs.
======================================================================================================================================================
A2. What is Apache Spark?

	An open-source, distributed, in-memory processing engine for large-scale data. It's "unified"
	because one engine handles: batch ETL, streaming, SQL analytics, machine learning (MLlib), and
	graph processing (GraphX) -- you don't need 5 separate tools.
	Written in Scala, runs on the JVM, exposes APIs in Python (PySpark), Scala, Java, R, and SQL.
	Current version: Apache Spark 4.0 (see A10).
======================================================================================================================================================

A4. Data Warehouse vs Data Lake vs Data Lakehouse

	DATA WAREHOUSE (DWH) -- e.g. Teradata, old-school SQL Server DW, Snowflake (sort of)
		+ Structured data only, strict schema, DML (insert/update/delete) is easy and fast, ACID.
		- Expensive to scale storage/compute independently, can't natively hold semi/unstructured
		  data (images, JSON logs, video), not built for ML/data-science workloads.

	DATA LAKE -- e.g. raw files in S3/ADLS/GCS (CSV, Parquet, JSON, images, video...)
		+ Cheap object storage, holds ANY data type, scales storage and compute independently.
		- No DML support out of the box (files are immutable blobs -- "updating a record" means
		  rewriting a whole file), no ACID guarantees (a failed write can leave data half-written/
		  corrupted), no schema enforcement, no time travel, poor performance for point queries.

	DATA LAKEHOUSE -- Databricks' pitch: best of both.
		A storage layer (Delta Lake, or increasingly Apache Iceberg) sits ON TOP of cheap data-lake
		object storage and adds: ACID transactions, schema enforcement + evolution, DML (update/
		delete/merge), time travel, and streaming support -- while still being cheap, scalable
		object storage underneath, and still able to hold any data type.
		In one line: a Data Lakehouse is an architecture that enables efficient, governed
		BI + AI directly on top of vast amounts of data sitting in a data lake, without a separate
		warehouse copy.

A5. Spark architecture -- how a job actually runs

	CLUSTER MANAGER -- allocates physical resources (CPU/RAM) to your Spark application across
	machines. On Databricks this is handled for you (you don't manage YARN/Kubernetes yourself
	unless you want to).

	DRIVER NODE -- the "brain." When you submit code (a notebook cell, a job), it runs on the
	driver. The driver:
		- builds the DAG (Directed Acyclic Graph) of the operations you asked for
		- splits the DAG into STAGES (a stage = a chunk of work with no shuffle boundary inside it)
		- splits each stage into TASKS (one task = one partition of data processed on one core)
		- schedules tasks onto executors and tracks their completion

	WORKER NODE -- a machine that runs one or more EXECUTORS.
	EXECUTOR -- a JVM process that actually runs tasks and holds cached data in memory/disk.
		Default: one or more executors per worker node; an executor caps out around 5 cores in
		practice (more cores per executor = more concurrent tasks, but also more chance of one
		executor causing memory pressure for everyone in it -- 5 is the empirically "sweet spot"
		Databricks and the community converged on).
		Rule of thumb: ONE core = ONE task = ONE partition being processed AT A TIME.

	EXECUTION FLOW:
		application submitted --> driver builds DAG --> DAG split into stages -->
		stage split into tasks (1 task per partition) --> tasks run in PARALLEL within a stage -->
		stage N+1 cannot start until stage N fully completes IF there's a data dependency
		(a "shuffle boundary") between them -- this is why wide transformations create new stages
		and narrow transformations don't.

	NOTE (2026): the description above is still exactly correct. What changed is that Adaptive
	Query Execution (AQE) now RE-PLANS the remaining stages after each stage finishes, using the
	actual output size/skew of the data rather than the original estimate. See C6.
======================================================================================================================================================
A6. Spark APIs: RDD vs DataFrame vs Dataset

	RDD (Resilient Distributed Dataset) -- the original, lowest-level API. A distributed collection
	of objects with no schema. "Resilient" = fault-tolerant (Spark can recompute lost partitions
	from lineage). Immutable, in-memory, distributed -- same as DataFrame on those three traits.

	DataFrame -- an RDD with a SCHEMA (like a relational table: named, typed columns). Because
	Spark knows the schema and the query, the CATALYST OPTIMIZER (A5/C6) can rewrite and optimize
	your query. RDDs get none of this optimization because Spark has no idea what's "inside" the
	opaque objects in an RDD.

	Dataset -- type-safe DataFrame (Scala/Java only, compile-time type checking). PySpark
	DataFrames are dynamically typed at the Python level, so "Dataset" isn't really a distinct
	concept you'll use in PySpark -- DataFrame is what you use.

	2026 PRACTICAL NOTE: you will essentially never touch raw RDDs on Databricks today. DataFrame/
	Spark SQL is the API for 99% of work; RDDs mostly show up in legacy code or very low-level
	custom partitioning logic.
======================================================================================================================================================
A7. Transformations vs Actions, Narrow vs Wide

	TRANSFORMATION -- returns a new DataFrame, LAZY (nothing actually executes yet).
		examples: select, filter/where, withColumn, groupBy, join, union, sort, drop, distinct

	ACTION -- triggers actual execution and returns a result (or writes output).
		examples: collect(), show(), count(), write.save(), first(), take(), foreach()

	Laziness matters because it lets Catalyst see your WHOLE chain of transformations before
	running anything, so it can reorder/combine/optimize the plan (e.g. push a filter down before
	a join instead of after).

	NARROW transformation -- each input partition contributes to at most ONE output partition.
	No shuffle needed. Examples: filter, map, union, withColumn.

	WIDE transformation -- data must move between partitions/executors ("shuffle") because the
	output depends on data spread across multiple input partitions. Examples: groupBy, join,
	distinct, repartition, orderBy/sort (global sort).
	Wide transformations are expensive (network I/O + disk spill) and are the main thing you tune
	for performance (Part C).
======================================================================================================================================================
A8. On-heap vs Off-heap memory

	ON-HEAP -- memory managed by the JVM garbage collector, lives inside the executor process.
		+ Faster than off-heap for typical workloads.
		- Garbage collection pauses can hurt performance/predictability when the heap fills up,
		  especially with many small objects.

	OFF-HEAP -- memory allocated outside the JVM heap, managed by the OS, not subject to GC.
		+ Better for very large datasets / avoiding GC pauses.
		- Slightly slower than on-heap due to serialization overhead; more I/O-bound.

	Off-heap is DISABLED by default. Enable with:
		spark.conf.set("spark.memory.offHeap.enabled", True)
		spark.conf.set("spark.memory.offHeap.size", "1g")
======================================================================================================================================================
A9. Databricks Compute -- 2026 model

	COMPUTE TYPES:
		- Serverless compute (DEFAULT/recommended in 2026) -- for notebooks, jobs, SQL warehouses,
		  and Lakeflow pipelines. Fully Databricks-managed, starts in seconds, SCALES TO ZERO when
		  idle, you pay per second of actual use. This has largely replaced the old advice of
		  "keep an all-purpose pool cluster warm so people don't wait for cold starts."
		- Classic all-purpose compute -- interactive cluster for notebook development, you manage
		  instance types/autoscaling. Still used when you need a specific instance type, init
		  scripts, custom libraries baked into the image, or long-running interactive dev sessions.
		- Classic job compute -- spun up for a scheduled job, torn down after -- cheaper than
		  all-purpose because it's not billed for idle interactive time.
		- Pools -- pre-warmed idle instances so a cluster can attach instantly instead of waiting
		  for cloud VMs to boot. Still exists, but serverless has eaten most of this use case.
		- Serverless GPU compute (new) -- fine-tuning / deep learning / LLM experimentation without
		  provisioning a GPU cluster yourself.

	CLUSTER MODES (unchanged concept):
		- Single node -- one machine acts as both driver and "worker" (no separate worker nodes).
		  Cheap, good for small workloads/dev/testing, not for heavy distributed jobs.
		- Multi node -- classic driver + N worker topology, for real distributed workloads.

	AUTOSCALING -- you set a min/max worker count; Databricks adds/removes workers based on load.
	Concept unchanged from your old notes; now largely superseded in day-to-day use by serverless
	compute's automatic scale-to-zero model.

B1. Reading a CSV file properly

	AVOID inferSchema=True in production. Reason: Spark makes TWO passes over the data --
	pass 1 reads the file to infer column names/types, pass 2 reads it again to build the actual
	DataFrame using that inferred schema. That's double the I/O for no reason if you already know
	the shape of your data.

	Correct approach -- define the schema explicitly and attach it:
		from pyspark.sql.types import StructType, StructField, StringType, IntegerType
		schema = StructType([
			StructField("id", IntegerType(), True),
			StructField("name", StringType(), True)
		])
		df = spark.read.format("csv").schema(schema).option("header", True).load("path")

	Job-count trivia (still true): if you specify NO schema info at all, Spark runs 1 job just to
	count columns. If you use inferSchema, Spark runs 2 jobs (columns, then types). If you attach
	an explicit schema, Spark runs 0 extra jobs just to read the file shape.

B5. Connecting to external cloud storage (ADLS / S3 / GCS)

	OLD PATTERN (your notes) -- dbutils.fs.mount() -- now LEGACY, avoid in new work:
		ACCESS_KEY = df.select('Access key ID').take(1)[0][0]
		SECRET_KEY = df.select('Secret access key').take(1)[0][0]
		ENCODED_SECRET_KEY = urllib.parse.quote(SECRET_KEY, safe="")
		SOURCE_URL = "s3a://%s:%s@%s" % (ACCESS_KEY, ENCODED_SECRET_KEY, BUCKET_NAME)
		dbutils.fs.mount(SOURCE_URL, "/mnt/mount_aws_s3/")

	Why it's discouraged now: a mount bakes a bucket-wide credential into the workspace with no
	fine-grained audit trail, and every user/job with cluster access effectively inherits it.

	NEW PATTERN (2026) -- Unity Catalog External Locations + Storage Credentials:
		CREATE STORAGE CREDENTIAL my_credential ...;
		CREATE EXTERNAL LOCATION my_location
		  URL 's3://my-bucket/path/'
		  WITH (STORAGE CREDENTIAL my_credential);

	Now access to that path is a governed, auditable Unity Catalog grant, just like a table grant
	-- you can see exactly who/what touched it in the audit log, and revoke access centrally.

B7. Null handling

	df.filter(df.emp_name.isNull())
	df.filter(df.emp_name.isNotNull())
	df.filter((df.emp_name.isNotNull()) & (df.salary > 3000))   -- always parenthesize each condition

	df.dropna()                          -- default behavior = "any": drops row if ANY column is null
	df.dropna(how="all")                 -- drops row only if ALL columns are null
	df.dropna(subset=["emp_name","salary"])  -- only check these columns

	df.fillna(value=0)                   -- fills only NUMERIC columns with 0 across the WHOLE df
	df.fillna(value="unknown")           -- fills only STRING columns with "unknown"
	df.fillna(value=0, subset=["col1","col2"])   -- restrict to specific columns

B8. UDFs (User Defined Functions)

	def my_func(x):
		return x * 2
	my_udf = udf(my_func, IntegerType())
	df = df.withColumn("doubled", my_udf(df.col1))

	UDFs are a "black box" to Spark's optimizer -- Catalyst has no visibility into what a UDF does
	internally, so it can't push filters through it, reorder around it, or vectorize it the way it
	does with built-in functions. Python UDFs specifically also pay a serialization cost moving
	data between the JVM and a separate Python process for every row/batch.
	Use UDFs when there's genuinely no built-in Spark function that does what you need -- otherwise
	prefer built-ins.

	2026 NOTE: this caveat got STRONGER, not weaker. Photon (Part C12), the vectorized native
	engine that now powers most compute by default, does NOT execute Python/Scala UDFs natively --
	hitting a UDF forces a fallback to plain JVM Spark for that stage, losing Photon's speedup for
	that portion of the query. This is now one of the most common reasons a "Photon-eligible" job
	doesn't actually run fast.

B9. String / Array functions

	split("colname", "-").getItem(0)     -- split on delimiter, 0 = first piece, 1 = second, etc.
	concat(col1, col2)                   -- glue strings together, NO delimiter: "a"+"b" -> "ab"
	concat_ws(",", col1, col2)           -- WITH delimiter: "a","b" -> "a,b"
	array_join(["a","b","c"], ",")       -- join array elements into a string: "a,b,c"

	arrays_zip("arr1","arr2")            -- pairs up elements from two arrays into one struct array
	                                         e.g. ([101,102],["HR","Fin"]) -> [{101,"HR"},{102,"Fin"}]
	explode(col)                         -- turns each array/map element into its own ROW
	array_intersect("a1","a2")           -- common elements, no duplicates
	array_union("a1","a2")               -- union, duplicates removed
	array_except("a1","a2")              -- elements in a1 NOT in a2
	array_distinct("a1")                 -- dedupe elements within one array
	array_min / array_max("a1")          -- min/max element value
	array_sort("a1")                     -- sort ascending; .desc() for descending
	array_repeat("col", 10)              -- repeats the value 10x into a list -- handy for
	                                         generating synthetic test data, then explode() it

B10. Aggregation / row-level helper functions

	-- Null count per column:
	from pyspark.sql.functions import col, count, when
	cols = df.columns
	null_counts = df.select([count(when(col(c).isNull(), c)).alias(c) for c in cols])

	df.agg({"Math": "max", "English": "min"})   -- COLUMN-level (aggregate) max/min

B11. Schema definition -- StructType, MapType, ArrayType

	StructType/StructField -- rigid, every declared field must have a value (or explicit null):
		schema = StructType([
			StructField("id", IntegerType(), nullable=True),
			StructField("name", StructType([
				StructField("firstName", StringType(), True),
				StructField("middleName", StringType(), True),
				StructField("lastName", StringType(), True)
			]))
		])
	Drawback: if a record is missing "middleName" entirely (not even null), you'll get an error
	building the DataFrame -- every declared field must be represented.

	MapType -- flexible alternative when a nested field's KEYS can vary or be absent:
		StructField("name", MapType(StringType(), StringType(), True))
	Now {"firstName":"a","lastName":"c"} (missing middleName entirely) does NOT error out.

	ArrayType -- when a column should hold a list of values:
		StructField("Hobbies", ArrayType(StringType()))

	Schema comparison between two DataFrames:
		df1.schema == df2.schema
		missing_or_extra = list(set(df1.columns) - set(df2.columns))

B13. dropDuplicates vs distinct

	df.dropDuplicates()           -- same as distinct() when called with no args
	df.dropDuplicates(["id"])     -- dedupe based on the "id" column ONLY, keeps all other columns

B14. select vs withColumn (performance)

	df.withColumn("full_name", concat(col("first"), lit(" "), col("last")))
		-- internally creates a whole new logical plan/projection node PER withColumn call;
		   chaining many withColumn calls in a loop is a known anti-pattern that slows down
		   planning (each one adds a projection to the plan tree).

	df.select("*", concat(col("first"), lit(" "), col("last")).alias("full_name"))
		-- does it in a single projection -- generally preferred, especially when adding MANY
		   columns at once.

	Both produce the same RESULT; select tends to win on planning-time performance, especially
	as the number of new columns grows.

B16. PySpark vs Pandas (and where Polars fits)

	PySpark                                          Pandas
	Python API, engine written in Scala/JVM          pure Python library
	distributed, multi-node processing               single-node only, no distributed processing
	DataFrames are immutable                         DataFrames are mutable
	supports true parallel processing                effectively single-threaded per operation
	LAZY evaluation (nothing runs until an action)    EAGER evaluation (runs immediately)

	Polars -- a newer Python DataFrame library (Rust-based) that's become popular since your notes
	were written. Single-node like Pandas, but far faster than Pandas due to a Rust core + lazy
	execution + multi-threading. Not distributed across a cluster like PySpark, so it's a
	single-node-fast tool, not a big-data-cluster tool -- think "great Pandas replacement," not
	"Spark replacement." Databricks Runtime supports installing/using Polars in notebooks like any
	other Python library.

	df1.subtract(df2)     -- rows in df1 not in df2, output is DE-DUPLICATED (no repeated rows)
	df1.exceptAll(df2)    -- rows in df1 not in df2, KEEPS duplicate rows (multiset semantics)

B18. Window functions -- lead/lag, first/last, dedup pattern

	from pyspark.sql import Window
	from pyspark.sql.functions import lead, lag, first, last, row_number, max as spark_max

	w = Window.partitionBy("id").orderBy("event_date")
	df.withColumn("next_val", lead("value", 1).over(w))
	df.withColumn("prev_val", lag("value", 1).over(w))
	df.withColumn("first_val", first("value").over(w))
	df.withColumn("last_val", last("value").over(w))

	-- classic "get max value among duplicates" interview pattern:
	w2 = Window.partitionBy("id")
	df.withColumn("max_price", spark_max("price").over(w2))
	df.withColumn("max_discount", spark_max("discount").over(w2))

	-- top/bottom N per group:
	w3 = Window.partitionBy("dept").orderBy(col("salary").desc())
	df.withColumn("rn", row_number().over(w3)).filter(col("rn") <= 3)   -- top 3 earners per dept

B19. Reading Excel files with multiple sheets

	df = spark.read.format("com.crealytics.spark.excel") \
	    .option("inferSchema", True).option("header", True) \
	    .option("dataAddress", "Sheet2!").load("path/to/file.xlsx")
	-- install com.crealytics:spark-excel from Maven at the cluster level first.

	The connector reads ONE sheet per call. To read ALL sheets and union them:

	def excel_df(path, sheets):
		df = None
		for sheet in sheets:
			sheet_df = spark.read.format("com.crealytics.spark.excel") \
			    .option("inferSchema", True).option("header", True) \
			    .option("dataAddress", f"{sheet}!").load(path)
			df = sheet_df if df is None else df.union(sheet_df)
		return df

B20. Skipping rows while reading a file

	spark.read.option("skipRows", 10).load(path)     -- skip first 10 rows

	-- to skip a MIDDLE range (e.g. rows 10-20), no direct option -- combine reads:
	full_df = spark.read.load(path)
	from_10  = spark.read.option("skipRows", 10).load(path)   -- everything from row 10 onward
	first_10 = full_df.subtract(from_10)                       -- isolates rows 0-9
	from_20  = spark.read.option("skipRows", 20).load(path)    -- everything from row 20 onward
	result = first_10.union(from_20)                            -- rows 0-9 + rows 20-onward
	                                                              -- i.e. rows 10-19 excluded

C1. Partitioning fundamentals

	Default read behavior: Spark splits input data into partitions of ~128MB each.
		spark.conf.get("spark.sql.files.maxPartitionBytes")   -- default: 134217728 bytes = 128MB
	Change it: spark.conf.set("spark.sql.files.maxPartitionBytes", <bytes>)

	df.rdd.getNumPartitions()          -- how many partitions the df currently has
	df.rdd.glom().collect()            -- see the actual DATA inside each partition (debug only,
	                                       never on large data -- it collects everything to the driver)

	sc.defaultParallelism              -- default parallelism = number of cores available in a
	                                       worker node (a starting point Spark uses for many defaults)

	Rough sizing table (block size 128MB default) -- table size vs target file size vs approx file count:
		10GB    -> 256MB target -> ~40 files
		1TB     -> 256MB target -> ~4,096 files
		2.56TB  -> 256MB target -> ~10,240 files
		3TB     -> 307MB target -> ~12,108 files
		5TB     -> 512MB target -> ~17,339 files
		7TB     -> 716MB target -> ~20,784 files
		10TB    -> 1GB target   -> ~24,437 files
	(Approximate guide for reasoning about "too many small files" problems -- in 2026, Liquid
	Clustering + Predictive Optimization (Part D21-22) handle most of this automatically for
	Delta tables, but the underlying math is still worth knowing for interviews.)

C2. repartition vs coalesce

	df.repartition(10, "country_id")      -- creates 10 EVEN partitions based on hashing country_id
	                                          REQUIRES a shuffle
	num = df.select("country_id").distinct().count()
	df.repartition(num, "country_id")     -- common pattern: size partitions to cardinality

	df.coalesce(5)                        -- REDUCES partition count to 5, NO shuffle (just merges
	                                          existing partitions together)

	KEY DIFFERENCE: repartition can INCREASE or DECREASE partition count (with a shuffle);
	coalesce can ONLY DECREASE partition count (without a full shuffle -- cheaper, but can create
	uneven partition sizes since it just glues existing partitions together rather than
	re-hashing everything).
======================================================================================================================================================
C3. Shuffle partitions

	spark.conf.get("spark.sql.shuffle.partitions")     -- default: 200
	spark.conf.set("spark.sql.shuffle.partitions", <n>)

	This is the number of partitions Spark creates whenever a wide transformation (groupBy, join,
	distinct, etc.) triggers a shuffle. Good practice: aim for ~128-200MB per shuffle partition,
	and a partition COUNT that's a multiple of your total core count so cores don't sit idle.

	2026 UPDATE: AQE (C6) now COALESCES shuffle partitions automatically after seeing the actual
	post-shuffle data size, so 200 behaves more like an UPPER BOUND / starting point than the
	literal number of partitions your job ends up using. You still set it, but you tune it less
	obsessively than you did on Spark 3.1.
======================================================================================================================================================
C4. partitionBy vs bucketBy (both apply only when WRITING data)

	partitionBy("year","month")   -- creates DIRECTORY-based partitions (a real folder per value):
		df.write.format("csv").option("header",True).option("maxRecordsPerFile",1000) \
		  .partitionBy("Year").mode("overwrite").save("path")
	Good for: columns you frequently FILTER or AGGREGATE on, with LOW-to-moderate cardinality
	(e.g. country, department, year). Bad for high-cardinality columns -- you get a folder PER
	distinct value, which can mean millions of tiny folders/files.

	bucketBy(5, "year","month").sort("product_name")   -- creates a FIXED number of buckets (5
	here) based on a HASH of the bucketing column(s), pre-sorted within each bucket.
	Good for: JOIN and GROUP-BY heavy workloads on HIGH-cardinality columns. When two tables are
	bucketed the same way on the join key, Spark can skip a full shuffle+sort at join time (data
	with the same key already lives in matching buckets) -- avoiding a costly Sort-Merge Join (C9)
	being recomputed from scratch on every query.

	KEY DIFFERENCE: partitionBy makes one folder PER DISTINCT VALUE (even a value with 1 record
	gets its own folder); bucketBy makes a FIXED number of buckets regardless of cardinality.

	2026 UPDATE -- Liquid Clustering (Part D21) is now the DEFAULT recommendation for new Delta
	tables instead of manually choosing partitionBy/bucketBy. Know partitionBy/bucketBy for
	legacy tables and for explaining WHY Liquid Clustering was invented (a very common interview
	framing: "what problem does Liquid Clustering solve that partitionBy/bucketBy had?").
======================================================================================================================================================
C5. Cache vs Persist vs Checkpoint

	CACHE -- stores a transformation's RESULT in MEMORY so repeated actions on it don't
	recompute the whole lineage from scratch.
		df.cache()          -- equivalent to df.persist(StorageLevel.MEMORY_ONLY)

	PERSIST -- like cache, but you choose WHERE it's stored:
		MEMORY_ONLY          -- memory only; if it doesn't fit, extra partitions are recomputed
		                         on demand instead of cached (not spilled to disk)
		MEMORY_ONLY_SER      -- same, but serialized (smaller memory footprint, slower access)
		MEMORY_AND_DISK      -- spills partitions that don't fit in memory to DISK instead of
		                         recomputing them
		MEMORY_AND_DISK_SER  -- same, serialized
		DISK_ONLY            -- disk only
		<LEVEL>_2 (e.g. MEMORY_ONLY_2)  -- replicates the cached data on 2 nodes, so losing one
		                                    node doesn't force a recompute

	Spark auto-cleans cached/persisted data when the application finishes successfully.
	Manual cleanup: df.unpersist()  -- there's no separate "uncache" command, unpersist covers both.

	CHECKPOINT -- writes the intermediate result of a DataFrame to DURABLE storage (e.g. HDFS/
	cloud storage), and -- critically -- TRUNCATES THE LINEAGE/DAG at that point.
	Example: if df6, df7, and df8 all depend on df5, and you call an action on any of them
	WITHOUT checkpointing df5, Spark recomputes df1->df2->df3->df4->df5 every single time. Checkpoint
	df5 once, and all three downstream branches read the saved result instead of recomputing.

	CACHE vs CHECKPOINT:
		- cache stores in memory (or disk via persist options); checkpoint ALWAYS writes to disk/
		  durable storage.
		- cache does NOT cut the DAG lineage (Spark still knows how to recompute if the cache is
		  evicted); checkpoint DOES cut the lineage -- the checkpointed data becomes the new
		  starting point, full stop.

		spark.sparkContext.setCheckpointDir("hdfs:///user/checkpoint_dir")
		df.checkpoint()
======================================================================================================================================================
C6. Catalyst Optimizer & Adaptive Query Execution (AQE) -- full picture

	WITHOUT AQE (Catalyst-only flow, the "static" plan):
		SQL / DataFrame / Dataset (front end)
		  --> Unresolved Logical Plan   (checks syntax + semantics -- do referenced tables/columns
		                                 exist?)
		  --> Logical Plan              (checks data types, applies rule-based optimizations --
		                                 e.g. arranging FROM/WHERE/GROUP BY correctly)
		  --> Optimized Logical Plan    (finds the best logical rewrite -- e.g. if filter A removes
		                                 10% of data and filter B removes 70%, apply B FIRST so less
		                                 data flows through the rest of the plan)
		  --> Physical Plan(s)          (there can be MANY valid physical plans for one logical plan)
		  --> Selected Physical Plan    (cost-model picks the best one out of the candidates)
		  --> executed as RDD operations (back end)

	The catch with Catalyst-only planning: the SELECTED physical plan is locked in and applied to
	EVERY stage of the job, using ESTIMATES made before any data was actually read (e.g. estimated
	table sizes from stale statistics).

	WITH AQE (default ON since Spark 3.2, more aggressive in Spark 4.0): Spark RE-EVALUATES the
	execution plan after EACH STAGE completes, using the stage's ACTUAL output size/shape --
	not the original estimate. This unlocks 3 automatic optimizations:
		1. Dynamically switching a shuffle join to a BROADCAST join if the actual (not estimated)
		   size of one side turns out to be small enough.
		2. Dynamically COALESCING many small post-shuffle partitions into fewer, right-sized ones
		   -- if you had 5 partitions and 3 turned out tiny + 2 turned out huge, without AQE the 3
		   small ones finish fast and their cores sit IDLE waiting on the 2 big ones; AQE merges
		   the small ones so cores stay busy.
		3. Automatically detecting and handling DATA SKEW at the join/aggregation level (splitting
		   an oversized partition rather than letting one task run 10x longer than its siblings).

	Enable manually (rarely needed, on by default): spark.conf.set("spark.sql.adaptive.enabled", True)

	INTERVIEW Q: "AQE already handles broadcast joins automatically -- why would you still
	broadcast a small dimension table explicitly?"
	A: for explicit CONTROL over the plan -- you don't want performance to depend on Spark's
	runtime size estimate being right; forcing it guarantees the behavior regardless of any future
	change to the data's actual size (and documents intent for the next engineer reading the code).
======================================================================================================================================================
C7. Broadcast joins & broadcast variables

	Normal join between a huge FACT table and a small DIMENSION table requires SHUFFLING data
	between nodes so matching keys land on the same executor -- expensive. This is sometimes
	called a "shuffle join."

	Instead, BROADCAST the small table: Spark ships a full COPY of the small table to every
	worker node, so the join happens LOCALLY on each executor with no shuffle needed.
	CAUTION: the broadcast table must fit comfortably in DRIVER memory (it's collected to the
	driver first, then pushed out to executors) and each executor's memory.

	from pyspark.sql.functions import broadcast
	df3 = df1.join(broadcast(df2), df1.id == df2.id, "inner")
	-- you will NOT see a shuffle/exchange step for df2 in the resulting execution plan.
	df3.explain(True)   -- True = show both logical AND physical plans; omit for physical plan only

	NOTE: if a dataset is under ~10MB, Spark will auto-apply a broadcast join even without you
	calling broadcast() explicitly (governed by spark.sql.autoBroadcastJoinThreshold).
======================================================================================================================================================
C8. Handling data skew

	Skew = data unevenly distributed across partitions (e.g. 80% of sales rows come from one
	country, 20% spread across 10 others -- partitioning naively by country dumps 80% of the work
	on a single executor while the rest sit mostly idle).

	1. Enable AQE (on by default from Spark 3.2+) -- from Spark 3.x onward, AQE automatically
	   handles salting, broadcast-join promotion, and skew-hint-style behavior for you in the
	   common case.
	2. Use a broadcast join if one side is small (C7).
	3. Salting (manual, for extreme skew AQE can't fully resolve) -- split an oversized key into
	   several pseudo-keys by appending a random "salt" value, spreading that key's rows across
	   more partitions, then aggregate back together in a second pass.
	4. Skew hint (manual) -- explicitly tell Spark a query is skewed:
	   spark.sql("SELECT * FROM mytable WHERE id > 100 /*+ SKEW('mytable','id') */")
	   (syntax varies by Spark version -- check current docs; the concept is "tell the optimizer
	   in advance which column/table is skewed so it applies mitigations without waiting to
	   discover it at runtime.")
======================================================================================================================================================
C9. Sort-Merge Join (SMJ)

	The default join strategy Spark uses internally for a standard (non-broadcast) join between
	two large DataFrames. Triggers automatically behind the scenes for inner/left/right/full joins
	when neither side is small enough to broadcast.

	Flow: SHUFFLE the data between executors (so matching keys land together) --> SORT each side
	by the join key --> MERGE the sorted sides together (single linear pass, like the "merge" step
	of merge-sort).

	This is exactly the expensive shuffle+sort pattern that bucketBy (C4) is designed to avoid --
	if both tables are already bucketed and sorted the same way on the join key at write time, the
	shuffle+sort step at query time can be skipped entirely.

	df.explain()             -- physical plan only (default)

C11. Delta Cache (a.k.a. disk cache)

	Delta tables physically live in remote object storage (DBFS/ADLS/S3/GCS). Every query against
	them, without caching, re-reads from that remote location -- adding network latency.
	Delta Cache copies frequently-accessed remote files onto the LOCAL DISK of the worker nodes
	running your queries, so subsequent reads hit local disk instead of remote storage.

	spark.conf.get("spark.databricks.io.cache.enabled")      -- disabled by default (on classic
	                                                              compute types where it's manual)
	spark.conf.set("spark.databricks.io.cache.enabled", True)
	CACHE SELECT * FROM delta_table_name        -- (conceptually similar to Snowflake's Result Cache,
	                                                but this caches the underlying FILES, not a
	                                                specific query's result set)

	2026 NOTE: on serverless compute, disk caching behavior is managed transparently by the
	platform (working alongside Photon) -- you spend a lot less time manually flipping this
	config than you did on classic clusters in 2023.

C12. Photon engine (new since your old notes)

	Photon is Databricks' native, vectorized query execution engine, written in C++ (not the JVM).
	Catalyst still PLANS the query (A5/C6 unchanged); Photon takes over EXECUTION for supported
	operators, processing data in COLUMNAR BATCHES of thousands of rows at a time instead of
	row-by-row -- this lets modern CPUs use SIMD instructions to process many values per cycle, and
	avoids JVM garbage-collection pauses and JIT warm-up entirely for the parts it handles.

	WHERE it's default-ON in 2026: serverless compute, all Databricks SQL warehouses, serverless
	Lakeflow pipelines, and classic all-purpose/job compute (opt-in checkbox "Use Photon
	Acceleration", or runtime_engine="PHOTON" via the Clusters/Jobs API, or photon=true via the
	Pipelines API).

	WHAT it accelerates: scans, joins, aggregations, window functions, sorts/limits, common string
	ops, AND writes (native Parquet writer speeds up Delta/Iceberg/Parquet writes including UPDATE/
	DELETE/MERGE INTO/INSERT/CTAS -- especially noticeable on wide tables with many columns).

	WHAT falls back to plain JVM Spark (loses the Photon speedup for that portion of the query):
		- Python and Pandas UDFs of any kind (see B8) -- the single most common cause
		- Scala UDFs (slightly better, but still triggers a fallback for that stage)
		- complex regex with non-literal patterns, lookahead/lookbehind
		- some MAP/STRUCT operations (transform_keys, transform_values, array aggregate) --
		  partially covered, varies by DBR version, check the query profile
	The fallback is TRANSPARENT (your query still returns correct results) -- it just silently
	loses the speed benefit for that stage, which is why "a job that's 80% Photon, 20% JVM ends
	up billed at the Photon DBU rate for the WHOLE run but only gets Photon-level speed on part
	of it" is a real gotcha worth knowing.
======================================================================================================================================================
C13. Predictive I/O (new)

	A Photon-layer read/write optimization: filter pushdown, dictionary pruning, and row-group
	skipping to avoid reading data from storage that a query doesn't actually need -- even when
	dealing with many small files. Works alongside dynamic file pruning (Spark automatically
	skipping whole files that can't possibly match a MERGE/UPDATE/DELETE's filter condition based
	on file-level statistics).

	Shuffle partition count (200)    | still a config, but AQE coalesces it post-shuffle -- you

PART D -- DELTA LAKE
======================================================================================================================================================
======================================================================================================================================================
D1. Why Delta Lake exists (recap, ties to A4)

	DWH: structured only, easy DML, cannot scale storage/compute independently, cannot hold semi/
	unstructured data.
	Data Lake: holds anything, cheap, scalable -- but NO DML support (data sits as raw immutable
	files, so "editing a record" means rewriting a whole file), no ACID (a failed write can leave
	data in a corrupted, half-written state), no schema enforcement.

	Delta Lake = a storage LAYER sitting on top of a data lake (the data lake still holds the raw
	files -- CSV/Parquet/JSON -- in cheap object storage). Delta Lake adds a metadata layer
	(column names/types/lengths/statistics) on top, which is what unlocks: easy DML, ACID
	transactions, time travel, schema enforcement/evolution, and streaming support. It combines
	the best of DWH (DML, structure, reliability) with the best of a data lake (scale, cost,
	flexibility) -- this is the core "Lakehouse" pitch (A4).
======================================================================================================================================================
D2. Delta Lake internal mechanism -- how a Delta table is actually stored

	When you create a Delta table, TWO things get written:
		1. ACTUAL DATA -- stored as Parquet files in your storage layer (the path you specify, or
		   a default managed location).
		2. METADATA / TRANSACTION LOG -- a "_delta_log" folder containing one JSON file PER
		   commit (every insert/update/delete/merge = one new JSON file, e.g. 000000.json,
		   000001.json, ...).

	Every 10 JSON commit files, Delta automatically writes a CHECKPOINT file in Parquet format
	that consolidates the log up to that point -- so time travel / reading current table state
	doesn't require replaying every single JSON file from the very first commit, just the latest
	checkpoint plus any JSON files after it.

	Alongside each JSON log file, Delta also writes a matching .crc (Cyclic Redundancy Check) file
	-- typically just a few bytes, computed with the CRC32 algorithm. It's used to (a) detect
	corruption -- if the .crc doesn't match its .json file, that json is treated as corrupted and
	NOT read, and (b) speed up queries -- the checksums help Delta quickly determine which log
	files it actually needs to read to satisfy a given query.

	DELETE behavior / retention: deleting data does NOT physically remove the underlying Parquet
	file immediately -- it's a SOFT delete (the file is marked invalid in the transaction log).
	Default retention: 7 days -- meaning you can time-travel back up to 7 days by default before
	the file becomes eligible for physical deletion (which only actually happens when VACUUM runs,
	Part D14). Change the retention window:
		SET delta.deletedFileRetentionDuration = "30 days";
	(longer retention = more storage used, since "invalid" files stick around longer.)

	Every DML operation on a Delta table is versioned -- each commit increments the table version,
	which is what powers time travel (D11).
======================================================================================================================================================
D3. Managed vs External (Unmanaged) tables

	MANAGED table -- Databricks manages BOTH the metadata AND the underlying data files for you
	(in a default or Unity-Catalog-governed storage location). This is the RECOMMENDED default
	approach -- you get the full benefit of time travel, ACID, and Unity Catalog security/
	governance features without thinking about file paths.

	EXTERNAL (unmanaged) table -- Databricks manages only the METADATA; you choose where the
	actual data files live (e.g. a specific S3/ADLS/GCS path you control). Gives the end user more
	control over the physical data location (useful for compliance/data-residency reasons or when
	another system also needs to read the raw files directly). Time travel and ACID still work.

	The core DIFFERENCE between managed and external is simply: WHO controls the physical file
	location, and what happens to the data files if you DROP the table (managed = files get
	deleted with the table; external = files remain, only the metadata pointer is removed).

	CREATE TABLE your_external_table
	USING parquet
	OPTIONS (
	  'path' 'abfss://acct@container.dfs.core.windows.net/path/to/parquet'
	);
======================================================================================================================================================
D4. Data Lake architecture vs Delta Lake (medallion) architecture

	PLAIN DATA LAKE architecture:
		source --> RAW (data lake) --> CURATED (data lake, basic cleansing/dedup) -->
		PRESENTATION/serving layer (data lake, aggregated/ready-to-use)
	All three layers sit in a plain data lake -- NO DML, NO ACID possible at any layer. You'd
	typically drive the transformations between layers with an external ETL tool (ADF, Azure
	Databricks jobs, AWS Glue, GCP Dataflow, etc.), and every "fix a bad record" scenario means
	rewriting whole files.

	DELTA LAKE (medallion) architecture -- the fix:
		source --> BRONZE (data lake) --> SILVER (delta lake) --> GOLD/serving layer (delta lake)
	Bronze is often still plain data-lake storage (its only job is to preserve raw history exactly
	as it arrived) -- Silver and Gold live in Delta Lake, unlocking DML/ACID/time travel from
	Silver onward. This is Databricks' recommended default pattern (see Part G for the full
	Bronze/Silver/Gold deep dive).
======================================================================================================================================================
D5. Ways to create a Delta table

	1. PySpark Delta API:
	   from delta.tables import *
	   DeltaTable.createIfNotExists(spark) \
	       .tableName("tab_name") \
	       .addColumn("emp_id", "INT") \
	       .property("some description") \
	       .location("some/path") \
	       .execute()
	   -- variants: .create(), .createIfNotExists(), .createOrReplace()

	2. SQL:
	   CREATE TABLE IF NOT EXISTS emp (
	     emp_id INT,
	     ...
	   ) USING DELTA
	   LOCATION 'some/path';
	   -- variants: CREATE TABLE, CREATE OR REPLACE TABLE

	3. From an existing DataFrame:
	   df.write.format("delta").option("path","some/location").saveAsTable("tab_name")

D6. Delta Table instance (for DML via PySpark instead of SQL)

	To run DML (update/delete/merge) via the PySpark API (rather than Spark SQL), first create an
	INSTANCE of the Delta table -- conceptually like getting a handle/reference to it, not a copy
	of the data. Operations on the instance still reflect on the underlying base table.

	from delta.tables import DeltaTable
	instance1 = DeltaTable.forPath(spark, "path/to/delta/table")
	instance2 = DeltaTable.forName(spark, "delta_table_name")
	-- now fire .update(), .delete(), .merge() etc. on instance1/instance2.

D7. Insert / Delete / Update -- the three ways to phrase each

	INSERT:
		%sql  INSERT INTO tab_name ...
		df.write.insertInto("tab_name", overwrite=False)
		spark.sql("INSERT INTO tab_name ...")
		df.write.mode("append").saveAsTable("tab_name")
		-- or: CREATE a VIEW from a source, then  INSERT INTO tab SELECT * FROM view

	DELETE:
		%sql  DELETE FROM emp WHERE ...
		DELETE FROM delta.`path/to/table` WHERE ...     -- less common, path-based
		spark.sql("DELETE FROM emp WHERE ...")

	UPDATE:
		same three styles as DELETE (SQL / path-based / spark.sql)

D8. SCD Type 1 via MERGE

	SCD1 = overwrite the old value with the new one, no history kept.
	Classic pattern (still 100% valid on Spark 4.0):
		MERGE INTO target t
		USING source s
		ON t.id = s.id
		WHEN MATCHED THEN UPDATE SET *
		WHEN NOT MATCHED THEN INSERT *

	2026 NOTE: inside a Lakeflow Declarative Pipeline specifically, you can now express this
	declaratively via AUTO CDC with STORED AS SCD TYPE 1 instead of hand-writing the MERGE
	(Part F12) -- but the raw MERGE pattern above is still exactly what you'd write in a normal
	(non-pipeline) notebook/job, and is still asked constantly in interviews.
======================================================================================================================================================
D9. Auditing a Delta table

	from delta.tables import DeltaTable
	instance = DeltaTable.forName(spark, "delta_emp")
	last_op = instance.history(1)         -- history(N) returns the last N commits' metadata

	You can explode the "operationMetrics" column from history() and insert selected fields into
	a dedicated audit table, alongside other useful audit columns you add yourself (username,
	notebook path, timestamp of the run, job/pipeline ID, etc.).
======================================================================================================================================================
D10. Slowly Changing Dimension Type 2 (SCD2)

	SCD2 = keep FULL HISTORY of changes -- when a dimension row changes, the OLD row is marked
	inactive (with an end_date and is_current=false) and a NEW row is inserted with the new
	values (start_date=now, is_current=true), instead of overwriting in place.

	Classic pattern via MERGE (two-pass: first expire changed rows, then insert new versions --
	or a single MERGE with WHEN MATCHED AND changed THEN ... / WHEN NOT MATCHED THEN INSERT,
	depending on your exact MERGE dialect's support for conditional multiple-match handling).

	2026 NOTE: Lakeflow Declarative Pipelines' AUTO CDC now supports STORED AS SCD TYPE 2 as a
	single declarative statement (Part F12) -- the engine manages start/end timestamps and the
	is_current flag for you, instead of you hand-writing the two-sided MERGE logic.
======================================================================================================================================================
D11. Time travel

	SQL:
		SELECT * FROM tab TIMESTAMP AS OF '2026-01-01T00:00:00'
		SELECT * FROM tab VERSION AS OF 12
		-- get valid timestamps/versions from: DESCRIBE HISTORY tab

	PySpark:
		df = spark.read.option("timestampAsOf", "2026-01-01T00:00:00").table("tab")
		df = spark.read.option("versionAsOf", 12).table("tab")

	Remember: you can only travel back as far as your retention window allows (default 7 days --
	D2), and only to versions whose underlying files haven't been physically removed by VACUUM.
======================================================================================================================================================
D12. RESTORE command

	After exploring old versions via time travel, if you decide to actually ROLL BACK the table to
	a previous state (not just read it), use RESTORE:
		RESTORE TABLE tab TO VERSION AS OF 12;
		RESTORE TABLE tab TO TIMESTAMP AS OF '2026-01-01T00:00:00';
	(PySpark equivalent: deltaTable.restoreToVersion(12) / .restoreToTimestamp("..."))
======================================================================================================================================================
D13. OPTIMIZE -- file compaction

	Every DML operation on a Delta table creates new files. Left unchecked, this "small file
	problem" grows over time (thousands of tiny files = expensive metadata management for Spark
	to plan around, and slow scans).

	OPTIMIZE combines small files into larger ones (target ~1GB by default). It does NOT delete
	the old small files immediately -- it marks them invalid in the transaction log and points the
	table forward to the new, combined file(s). (Physical removal of the old invalid files is
	VACUUM's job -- D14.)
		OPTIMIZE delta_tab_name;

	2026 NOTE: Predictive Optimization (D22) now runs this automatically in the background on
	Unity Catalog managed tables -- manual OPTIMIZE is the fallback, not the default workflow.
======================================================================================================================================================
D14. VACUUM -- cleanup of invalid files

	Walkthrough of how "invalid" files accumulate:
		- insert 4 records -> 4 new Parquet files created
		- delete record 1 -> that file is marked INVALID (not physically deleted, no new file
		  created for a pure delete)
		- update record 2 -> that file is marked INVALID, and 2 NEW files are created (the old
		  file's other still-valid rows + the updated row get rewritten)
		- insert a new record -> 1 new file created
		- run OPTIMIZE -> combines all currently-valid smaller files into one big (<=1GB) file,
		  and marks ALL those smaller files invalid going forward

	Invalid files just keep accumulating with every DML + OPTIMIZE cycle unless you clean them up
	-- that's what VACUUM does: PHYSICALLY delete files that have been invalid for longer than the
	retention period.

	ALWAYS dry-run first:
		VACUUM delta_tab_name DRY RUN;                       -- lists files that WOULD be deleted
		VACUUM delta_tab_name RETAIN 720 HOURS DRY RUN;       -- only consider files invalid > 720h (30 days)
		VACUUM delta_tab_name RETAIN 0 HOURS DRY RUN;         -- errors by default (Databricks safety check)

	To force deletion of everything invalid RIGHT NOW (dangerous -- kills time travel for those
	versions), disable the safety check first:
		SET spark.databricks.delta.retentionDurationCheck.enabled = False;
		VACUUM delta_tab_name RETAIN 0 HOURS;

	Default: VACUUM removes files that have been invalid for more than 7 days.
	**IMPORTANT: after VACUUM runs, you CANNOT time-travel back to versions whose files it removed.**
======================================================================================================================================================
D15. Z-ORDER

	An EXTENSION of OPTIMIZE. Plain OPTIMIZE combines small files into bigger ones, but doesn't
	control what ORDER the data sits in inside those combined files -- so an analytical
	filter/aggregation/join might still scan files it didn't need to.

	Z-ORDER sorts and co-locates related data together (by one or more columns) WHILE combining
	files, so filters/joins/aggregations on the Z-ordered column(s) only need to scan the specific
	files containing relevant data, skipping the rest entirely.

		OPTIMIZE delta_emp_type2 ZORDER BY (emp_id);

	2026 NOTE: Liquid Clustering (D21) has effectively replaced Z-ORDER + partitionBy as the
	go-to layout technique for NEW tables -- it achieves a similar "co-locate related data"
	outcome, but incrementally and without you needing to re-run a manual ZORDER command
	periodically.
======================================================================================================================================================
D16. Schema evolution -- mergeSchema vs overwriteSchema

	Real-world schemas drift over time (e.g. a table starts as id/name/salary, and months later
	the source starts sending an extra "dept" column too).

	mergeSchema=True (ADD columns automatically):
		df.write.option("mergeSchema", True).format("delta").mode("append").save(path)
	-- new incoming columns get added to the Delta table automatically; OLD existing records get
	   NULL for the newly-added column (they never had a value for it).
	-- if a column DISAPPEARS from the source, mergeSchema does NOT drop it from the Delta table
	   -- it just keeps inserting NULL for that column going forward for new records.

	overwriteSchema=True (REPLACE the schema entirely):
		df.write.option("overwriteSchema", True).format("delta").mode("overwrite").save(path)
	-- this WILL drop/rename columns to exactly match the incoming source's current schema --
	   use with caution, this is a destructive schema change.

	2026 NOTE: Delta Lake 4.x adds TYPE WIDENING (D24) -- e.g. INT -> LONG, FLOAT -> DOUBLE -- as
	a supported schema-evolution operation that does NOT require a full table rewrite, which
	mergeSchema alone did not cover in your old notes' era.
======================================================================================================================================================
D17. Column reordering for query performance

	Why it matters: when you write to a Delta table, Delta captures column-level STATISTICS
	(min/max values etc.) in the JSON transaction log to enable data-skipping optimizations --
	BUT it only captures stats for the FIRST 32 columns by default. If your table has 100+
	columns and the columns you actually FILTER/query on most often sit past column 32, you lose
	the benefit of data skipping for them.

	Fix: reorder so your most-queried columns are within the first 32:
		ALTER TABLE emp CHANGE COLUMN col35 FIRST;
		ALTER TABLE emp CHANGE COLUMN col35 AFTER col10;
	(Opening the transaction log's JSON files afterward, you'll see stats now being captured for
	col35.) Note: the reorder only affects NEW data written AFTER the change -- existing data
	already on disk doesn't get physically rewritten just because you reordered columns in the
	schema; it takes effect as new records are loaded going forward.
======================================================================================================================================================
D18. Cloning Delta tables

	DEEP CLONE -- copies the DATA physically (a real, independent copy of all files). Slower,
	uses more storage, but the clone is fully independent of the source going forward. Typically
	used for backups / point-in-time snapshots you want to keep safe even if the source is dropped.

	SHALLOW CLONE -- copies only the METADATA (the transaction log points at the SAME underlying
	data files as the source). Fast, cheap on storage, but the clone depends on the source's data
	files still existing -- good for quick "test against a copy" scenarios, not for durable backups.
======================================================================================================================================================
D19. Types of views

	NORMAL (persistent) view -- persists across sessions in the catalog/metastore; must be
	explicitly dropped to remove it.

	TEMP view -- exists for the current SESSION ONLY (tied to the notebook/session -- gone when
	the session ends, e.g. cluster restart, detach/reattach). NOT persisted in Unity Catalog, and
	not directly browsable from Unity Catalog's UI -- but usable inside SQL queries/notebooks
	within that same session.

	GLOBAL TEMP view -- exists at the CLUSTER level, visible to ANY session/user attached to that
	same cluster. Dropped when the cluster restarts. Stored in a special global-temp schema and
	IS visible via SHOW VIEWS / manageable via SQL commands in that schema.
======================================================================================================================================================
D20. Reading files directly via SQL (no DataFrame needed)

	SELECT * FROM json.`a/bc/file1.json`;
	SELECT * FROM parquet.`path/to/file.parquet`;
	SELECT * FROM csv.`path/to/file.csv`;
	-- format-name-as-a-"database", backtick-quoted path as the "table" -- handy for one-off
	   ad hoc exploration without formally registering a table.
======================================================================================================================================================
D21. LIQUID CLUSTERING (new, 2026's default recommendation)

	Replaces partitionBy/bucketBy (D4/C4) as the primary data-LAYOUT tuning lever for NEW Delta
	tables.
		CREATE TABLE sales.transactions
		CLUSTER BY (customer_id, transaction_date);

	Why it's better than the old approach:
		- No fixed partition scheme to get wrong up front -- Databricks incrementally re-clusters
		  data as it lands, rather than requiring you to pick the "right" partition columns before
		  you've seen how the data actually behaves.
		- Clustering keys can be CHANGED LATER without a full table rewrite -- partitionBy
		  fundamentally could NOT do this (changing partition columns on an existing partitionBy
		  table meant rewriting the whole table).
		- Solves exactly the "high-cardinality partitionBy column creates millions of tiny
		  folders" problem your old notes describe.

	Interview framing: "what problem does Liquid Clustering solve that partitionBy had?" --
	answer with the two bullets above (rigid upfront choice + can't change without a rewrite).
======================================================================================================================================================
D22. PREDICTIVE OPTIMIZATION (new)

	On Unity Catalog MANAGED tables, Databricks can now run OPTIMIZE / VACUUM / ANALYZE
	AUTOMATICALLY in the background, using a model to decide WHEN a given table actually needs
	compaction/stats-refresh -- instead of you scheduling "run OPTIMIZE every Sunday night"
	regardless of whether the table actually changed enough to need it that week.
	Manual OPTIMIZE/VACUUM/ANALYZE still exist and still work exactly as documented in D13-D14 --
	they're now the FALLBACK for edge cases (e.g. an external table Predictive Optimization
	doesn't cover), not the default operating model for managed tables.
======================================================================================================================================================
D23. DELETION VECTORS (now default-on for most workloads)

	Historically (and still true for tables WITHOUT deletion vectors enabled): a DELETE/UPDATE/
	MERGE on a Delta table rewrites the ENTIRE Parquet file(s) containing the affected rows, even
	if you're only touching a handful of rows out of millions in that file.

	Deletion vectors change this: instead of rewriting the whole file, Delta writes a small
	"soft-delete marker" file that records WHICH ROWS in an existing Parquet file are logically
	deleted/updated, without touching the original file at all. Readers transparently apply the
	deletion vector when scanning, so query results are still correct -- but the WRITE itself is
	dramatically cheaper for row-level DML on large tables.
	This changes the "every DELETE rewrites files" mental model from earlier Delta versions --
	worth explicitly correcting if you're used to describing DELETE that way from older training/notes.
======================================================================================================================================================
D24. Delta Lake 4.x feature additions

	- VARIANT columns natively supported in Delta tables (ties to Spark 4.0's VARIANT type, B22).
	- Coordinated commits -- allows MULTIPLE ENGINES (not just Spark) to safely write to the same
	  Delta table concurrently, without corrupting the transaction log.
	- Type widening (mentioned in D16) -- e.g. INT->LONG, FLOAT->DOUBLE -- without a full table
	  rewrite or a Lakeflow pipeline reset.
	- Catalog-managed tables -- tighter integration where the catalog (Unity Catalog) is
	  authoritative over table metadata rather than the table "owning" its own log independently.
======================================================================================================================================================
D25. UniForm -- Delta <-> Iceberg <-> Hudi interoperability (GA)

	Delta Universal Format (UniForm) makes Delta AUTOMATICALLY write Apache Iceberg (and Apache
	Hudi) METADATA alongside its native Delta transaction log -- while the underlying DATA files
	stay as plain Parquet (which all three formats are ultimately thin metadata layers on top of
	anyway).

	Result: engines that only understand Iceberg (Snowflake, BigQuery, Redshift, Athena, Trino,
	etc.) can read the EXACT SAME physical files as a native Iceberg table would -- with NO copy,
	NO separate conversion job, and negligible extra write overhead (UniForm is "only writing
	additional metadata"). UniForm is compatible with Liquid Clustering too -- when Liquid
	Clustering reorganizes data, UniForm reflects that improved layout in BOTH the Delta and the
	Iceberg metadata.

	Practical 2026 answer to "Delta or Iceberg, which do we pick?" -- often you don't have to
	choose exclusively: pick based on your PRIMARY engine ecosystem (Databricks-native workloads
	-> Delta; multi-engine-first shops -> Iceberg), and let UniForm bridge the gap for whichever
	other engines need to read the data too.
======================================================================================================================================================
D26. Iceberg v3 GA + Managed Iceberg tables in Unity Catalog

	Unity Catalog now natively GOVERNS Apache Iceberg tables too, not just Delta tables -- meaning
	you can create a "Managed Iceberg" table in Unity Catalog with the SAME governance model
	(permissions, lineage, audit) you already use for Delta tables.
	Iceberg v3 (the spec itself) added deletion vectors, row lineage, and a VARIANT-equivalent
	type -- closing what used to be a real feature gap between Delta and Iceberg as of your old notes.
======================================================================================================================================================
D27. New column types (2026)

	GEOSPATIAL types (GA, in both Delta and Iceberg v3) -- native geometry/geography support for
	use cases like route optimization, fleet/logistics analysis, geofenced risk monitoring --
	previously you'd have needed a separate specialized GIS/spatial database for this.

	FILE type (Beta) -- lets a managed Delta or Iceberg table natively GOVERN unstructured data
	(PDFs, images, audio, video) as a proper column type alongside your normal structured columns
	-- instead of storing unstructured files separately (e.g. in a Volume, D-something) and only
	linking to them by path/reference from a structured table.

======================================================================================================================================================
E1. What Unity Catalog is, and why it exists

	Unity Catalog (UC) is Databricks' fine-grained GOVERNANCE layer for everything on the
	lakehouse -- tables, views, volumes (files), ML models, functions, and (as of 2026) AI
	models/agents/tools too (Part E10 onward, and see the AI notepad's Unity AI Gateway section).
	It provides ONE central place to administer and audit ALL data/AI access, instead of managing
	permissions separately per-workspace or per-cluster the way the old Hive Metastore model did.

	Core value propositions (straight from Databricks' own framing, still accurate in 2026):
		- UNIFIED CATALOG -- one catalog stores your data, ML models, and analytics artifacts,
		  plus metadata for each object, and can blend in data from other catalogs (e.g. an
		  existing Hive Metastore during migration).
		- UNIFIED ACCESS CONTROL -- a single, consistent permissions model across ALL data assets
		  and ALL clouds, including attribute-based access control (ABAC) for handling PII.
		- DATA ISOLATION -- isolate at multiple levels (environment / storage location / individual
		  object) without losing centralized manageability and auditing.
		- DATA AUDITING -- every access is centrally logged (queryable via the event_logs /
		  system tables) with alerting/monitoring, including who created/altered/dropped objects.
		- DATA QUALITY MANAGEMENT -- built-in quality controls/testing/monitoring/enforcement.
		- DATA LINEAGE -- end-to-end visibility into how data flows through the lakehouse, source
		  to consumption.
		- DATA DISCOVERY -- lets data scientists/analysts/engineers quickly find and reference
		  relevant data.
		- DATA SHARING -- share Delta Lake objects (tables, views, notebooks) across regions,
		  platforms, users, and cloud providers via the open Delta Sharing protocol.
======================================================================================================================================================
E2. Setting up Unity Catalog (the actual admin steps)

	1. Create a storage account (Blob or ADLS) -- this holds the metastore's metadata storage.
	2. Create a Databricks connector/access-connector service in Azure (or the equivalent
	   IAM role/service principal on AWS/GCP).
	3. Grant that connector the needed access (e.g. "Storage Blob Data Contributor") on the
	   storage account from step 1.
	4. In Databricks account admin settings, create the Unity Catalog metastore and ATTACH the
	   workspace(s) you want governed by it.

	IMPORTANT CONSTRAINTS to remember for interviews:
		- You can have one Unity Catalog METASTORE per REGION.
		- A metastore created in Region X cannot be assigned to a workspace in Region Y.
		- Each WORKSPACE can be attached to only ONE metastore.
		- ONE metastore CAN be attached to MULTIPLE workspaces (that's how you get consistent
		  governance across, e.g., a dev workspace and a prod workspace in the same region).
		- You do NOT need to separately create a SQL endpoint/warehouse just for UC access -- as
		  long as the compute (cluster/warehouse) itself has UC access enabled, it can query
		  UC-governed objects.
======================================================================================================================================================
E3. The 3-level namespace

	OLD (Hive Metastore era): database.table   (two levels)
	NEW (Unity Catalog): catalog.schema.table   (three levels)
	e.g. sales_prod.orders.transactions
	The extra "catalog" level lets you cleanly separate, e.g., environments (dev/staging/prod
	catalogs) or business domains, all within one metastore, with permissions grantable at any of
	the three levels (catalog-wide, schema-wide, or table-specific).
======================================================================================================================================================
E4. Volumes (new UC object type)

	Volumes govern NON-tabular files (PDFs, images, model weights, arbitrary binary blobs) with
	the SAME grant/audit/discovery model Unity Catalog already applies to tables.
	Modern notebooks reference files via a Volume path like:
		/Volumes/<catalog>/<schema>/<volume_name>/path/to/file
	instead of a raw, ungoverned DBFS path -- this is the direct replacement for "just drop files
	in DBFS and hope nobody misuses the path" from the pre-UC era.
======================================================================================================================================================
E5. Lineage

	Automatic, COLUMN-LEVEL lineage is captured across notebooks, jobs, and dashboards --
	you can see exactly which upstream table/column a downstream table/column/dashboard field was
	derived from, without manually documenting it.
	EXTERNAL LINEAGE (GA, Summit 2026) extends this graph BEYOND Databricks -- you can register
	upstream SOURCE systems and downstream BI tools (Tableau, Power BI, etc.) so a single lineage
	graph spans your FULL data flow end-to-end, not just the Databricks-internal portion of it.
	Lakeflow Connect ingestion pipelines now automatically record this lineage from source tables

E6. External Locations & Storage Credentials (replaces mounts -- see also B5)

E7. ABAC (attribute-based access control) + cross-engine enforcement

	non-Databricks engine querying via Catalog Federation (E8) or an external reader.

E8. Catalog Federation (new)

	Query data that physically lives in SNOWFLAKE, BIGQUERY, or Microsoft ONELAKE/Fabric DIRECTLY
	through Unity Catalog -- no copy, no ETL pipeline needed to "bring the data into Databricks"
	first. Azure Databricks <-> OneLake federation is bidirectional and GA as of Summit 2026 --

	physically IN OneLake (Public Preview) so Microsoft Fabric engines can read them with zero copy too.

E9. Unity Catalog Metrics -- semantic/KPI layer (new, Summit 2026)

	Define a business KPI (e.g. "revenue," "churn," "active users") ONCE as a governed, reusable
	object, with the underlying joins/filters/time-grain baked into the definition. SQL queries,
	BI dashboards, and AI agents can all reference the SAME metric definition instead of each team
	re-deriving "revenue" slightly differently (the classic "marketing's number doesn't match
	finance's number" problem).
	Supports: multi-fact relationships (metrics spanning more than one fact table), level-of-detail
	(LOD) calculations (compute at whatever grain you specify, not just the query's grain),
	parameterized metrics (adapt based on runtime inputs), and window measures (period-over-period
	comparisons made easy). Open-sourced: available in Apache Spark and Unity Catalog OSS, and
	built to be Open Semantic Interchange (OSI) compatible with other tools' semantic layers.

E11. OpenSharing (evolution of Delta Sharing)

	Delta Sharing was Databricks' original open protocol (2021) for secure, ZERO-COPY data sharing
	across organizations or teams, regardless of what compute platform the RECIPIENT uses --
	you can share Delta Lake objects (tables, views, notebooks) across regions/platforms/users/

	Steps to share (region1 -> region2 example, unchanged from your old notes):
		1. Unity Catalog > Delta Sharing > Create Recipient

======================================================================================================================================================
F1. Structured Streaming basics (foundation, unchanged)

	READ:
		stream_df = spark.readStream.table("source_table")
		# or: spark.readStream.format("...").load("path")

	WRITE:
		stream_df.writeStream \
		    .trigger(processingTime="5 minutes") \
		    .outputMode("append") \
		    .option("checkpointLocation", "/path/to/checkpoint") \
		    .table("target_table")

	TRIGGER options:
		.trigger(processingTime="5 minutes")   -- check the source for new data every 5 minutes
		                                           (default, if unspecified: every ~30 seconds)
		.trigger(once=True)                    -- process ALL currently-available data ONE TIME,
		                                           then STOP (classic "batch-ify a stream" pattern)
		.trigger(availableNow=True)            -- process all currently-available data in
		                                           MICRO-BATCHES, then STOP (more efficient than
		                                           once=True for large backlogs)

	OUTPUT MODE options:
		.outputMode("append")     -- append only newly-arrived records
		.outputMode("complete")   -- overwrite the ENTIRE target with the full current result
		                              every trigger (used for aggregation queries where the whole
		                              result can change, not just new rows)

	CHECKPOINTING -- .option("checkpointLocation", "/path") tracks progress (which offsets/files
	have already been processed), which is what guarantees EXACTLY-ONCE processing semantics and
	fault tolerance (if the stream crashes and restarts, it resumes from the checkpoint instead of
	reprocessing everything or skipping data).

	KNOWN LIMITATIONS of streaming DataFrames: you CANNOT sort a streaming DataFrame (no global
	SORT support without bounding the data first), and you cannot straightforwardly DEDUPLICATE
	across the whole unbounded stream (only within a bounded window, using watermarking).
======================================================================================================================================================
F2. Auto Loader basics

	Auto Loader is built on the Structured Streaming concept above, specialized for incrementally
	and efficiently picking up NEW FILES as they land in cloud storage.

	READ:
		stream_df = spark.readStream \
		    .format("cloudFiles") \
		    .option("cloudFiles.format", "csv") \
		    .load("path/to/files")

	WRITE: identical writeStream pattern as F1.
======================================================================================================================================================
F3. COPY INTO vs Auto Loader

	COPY INTO is a simpler, SQL-based, idempotent bulk-load command:
		COPY INTO table_name FROM 'file/path' FILEFORMAT = CSV;
	(re-running it is safe -- it only loads files it hasn't already loaded)

	WHEN TO USE WHICH:
		COPY INTO     -- good for up to low THOUSANDS of files; simpler, less efficient at scale
		Auto Loader   -- built for MILLIONS of files; more efficient at scale, streaming-native,
		                 supports both directory-listing and cloud-notification-based file detection
======================================================================================================================================================
F4. Medallion / multi-hop architecture -- code walkthrough

	SOURCE --> BRONZE --> SILVER --> GOLD

	Source to Bronze (raw ingestion, minimal/no transformation):
		stream_df = spark.readStream.format("cloudFiles") \
		    .option("cloudFiles.format", "csv").load("file/path")
		stream_df.writeStream.trigger(processingTime="5 minutes") \
		    .outputMode("append").option("checkpointLocation","/path").table("bronze_table")

	Bronze to Silver (cleansing, type casting, add tracking columns like input_file_name):
		spark.readStream.table("bronze_table").createOrReplaceTempView("bronze_view")
		-- apply transformations against bronze_view, write result onward to silver

	Silver to Gold (aggregations, business logic):
		-- build a temp view with aggregations over the silver view (e.g. sum(sales) grouped by date/city)
		spark.table("silver_agg_temp_view").writeStream \
		    .outputMode("append").option("checkpointLocation","/path").table("gold_table")
======================================================================================================================================================
F5. Auto Loader incremental load -- full mechanics

	INCREMENTAL LOAD characteristics (what you WANT):
		1. process only NEW or MODIFIED files
		2. process as soon as possible after new data is detected
		3. never MISS a new file
		4. stay performant even against very LARGE directories
		5. be a REPEATABLE pattern (safe to re-run/restart)

	Older, worse patterns and why they fall short:
		- Watermark pattern (store a timestamp somewhere, compare on each run) -- the pipeline
		  still has to run on a SCHEDULE whether or not new data actually arrived, wasting cost.
		- Plain checkpointing without Auto Loader -- hits a performance bottleneck LISTING very
		  large directories with millions of files (a full directory listing every run is slow).

	AUTO LOADER solves this by continuously, efficiently, and automatically loading new files as
	they arrive, rather than re-scanning everything each run.

	What it supports:
		- Cloud storage: ADLS, S3, GCS, DBFS
		- File formats: JSON, CSV, Parquet, AVRO, ORC, TEXT, and binary files

	Internal components:
		- stores metadata of already-processed files in an internal RocksDB-backed database (so
		  it always knows what it's already ingested, without a full directory re-listing)
		- processes new files immediately via Spark Structured Streaming
		- uses optimized cloud-native components to detect new files as soon as they arrive
	Auto Loader = Cloud Files Data Reader + Cloud Notification Services, combined.

	FILE DETECTION MODES:
		1. Directory listing mode (DEFAULT if you specify nothing) -- periodically lists the
		   target directory to find new files. Simpler to set up, works everywhere.
		2. File notification mode (FASTER, but needs more setup) -- subscribes to native cloud
		   storage event notifications (e.g. Azure Event Grid / AWS S3 event notifications) so new
		   files are detected the moment they land, instead of waiting for the next listing pass.
		   Requires you to configure a service principal (subscription id, connection string,
		   tenant ID, client ID, client secret, resource group).

	Basic config (directory listing mode, implied by not specifying notifications):
		spark.readStream.format("cloudFiles").option("cloudFiles.format","csv").load("path")

	With notifications enabled:
		spark.readStream.format("cloudFiles") \
		    .option("cloudFiles.useNotifications", True) \
		    .options(**cloud_file_config) \    # subscriptionId, connectionString, tenantId,
		                                        # clientId, clientSecret, resourceGroup
		    .option("cloudFiles.format", "csv") \
		    .load("path")

	To write a FULL LOAD (one-time catch-up) instead of continuous micro-batches:
		stream_df.writeStream.trigger(once=True).save("table/location")
		# use .trigger(processingTime="1 minute") etc. instead for ongoing incremental loads

F6. Why declarative pipelines beat hand-written procedural ETL

	Building a data pipeline "by hand" (procedural approach) generally requires you to personally
	handle EVERY one of: code development, cluster management, data quality checks, infrastructure
	management, dependency management, fault tolerance, data governance, and promotion to production.

	DLT / Lakeflow Declarative Pipelines take everything EXCEPT code development off your plate --
	you write WHAT the tables should look like, and the framework handles cluster provisioning,
	dependency ordering, retries/fault tolerance, and (with expectations) data quality enforcement.

	PROCEDURAL approach:
		1. explicitly define every ETL step (ingest, transform, load) yourself
		2. steps must follow a specific order YOU maintain
		3. you dictate exactly HOW each step should be performed
		4. lots of code, time-consuming, error-prone
		5. debugging/error-handling is a real challenge

	DECLARATIVE approach (opposite in every way):
		1. all ETL mechanics are abstracted away from you
		2. no specific order for YOU to maintain -- the engine figures out dependency order
		3. you declare WHAT is needed, not HOW to get there
		4. the engine automatically generates the extract/transform/load code/execution plan
		5. much less code to write and maintain
		6. debugging/error-handling is meaningfully easier (built-in observability)
	(Think of it as "outsourcing the procedural plumbing to the platform.")

	Great functionalities Lakeflow Declarative Pipelines handles automatically for you:
		data transformation, cluster management, data quality validation, accelerated
		development, task orchestration, monitoring, error handling, and unified batch/streaming
		processing in one pipeline (a genuinely hard problem to hand-roll -- e.g. loading
		streaming data into a fact table while loading slower-changing batch data into a
		dimension table, in the SAME pipeline).

	Development steps:
		1. write live tables/pipeline code in a notebook (or Lakeflow Designer, F14, no-code)
		2. create a pipeline (previously "via Workflows," now "via Lakeflow Jobs," F15)
		3. run/schedule the pipeline
		4. monitor metrics in the pipeline UI

	Note: your old notes call this "DLT." As of 2025-2026, it's Lakeflow Declarative Pipelines --
	same product, renamed and expanded (Part F9 onward covers exactly what changed).

F7. Dataset types -- Streaming Table / Materialized View / View

	IMPORTANT: pipelines currently support Python and SQL as the authoring languages.

	1. STREAMING TABLE
		- supports incremental/streaming ingestion from APPEND-ONLY sources; each record is
		  processed EXACTLY ONCE.
		- Auto Loader is typically used underneath to detect/process new files individually as
		  they arrive.
		- good fit for: high-volume, rapidly-growing, low-latency data; no re-computation of
		  already-processed old data needed. Classic BRONZE-layer dataset type.

		SQL:
			CREATE STREAMING TABLE customers
			AS SELECT * FROM cloud_files('mnt/a/b/', 'csv');

		Python (OLD DLT syntax -- still runs):
			import dlt
			@dlt.table
			def customers():
			    return (spark.readStream.format("cloudFiles")
			        .option("cloudFiles.format", "json")
			        .option("cloudFiles.inferSchema", True)
			        .load("mnt/a/b/"))

	2. MATERIALIZED VIEW (formerly also called a "Live Table" under the DLT name)
		- represents the CURRENT STATE of a defined query's result -- great for handling CDC-style
		  scenarios (insert/update/delete all reflected correctly).
		- the result is stored as a table, but you don't run DML on it manually -- ALL changes to
		  it flow purely from re-evaluating the DEFINING QUERY.
		- best fit for: transformations/aggregations (classic SILVER and GOLD layer dataset type).

		SQL:
			CREATE MATERIALIZED VIEW sales_agg AS
			SELECT order_dt, city, product, SUM(sales_amount) AS total_sales
			FROM sales s JOIN customers c ON s.cust_id = c.id
			WHERE product = 'laptop'
			GROUP BY order_dt, city, product;

	3. VIEW
		- represents the current state of a defined query, but the DATA IS NOT PERSISTED.
		- only referenceable WITHIN the pipeline that defines it (not queryable from outside).
		- best fit for: intermediate results the end user/downstream apps never need to see directly.

		SQL:
			CREATE VIEW v_sales_agg AS
			SELECT order_dt, city, product, SUM(sales_amount) AS total_sales
			FROM sales s JOIN customers c ON s.cust_id = c.id
			WHERE product = 'laptop'
			GROUP BY order_dt, city, product;

	2026 SYNTAX UPDATE (Lakeflow Declarative Pipelines' new Python API):
		OLD:  import dlt              NEW:  from pyspark import pipelines as dp
		      @dlt.table                     @dp.table                 -- streaming tables
		                                      @dp.materialized_view     -- NEW, explicit decorator
		                                                                   for materialized views
		                                                                   (old @dlt.table covered
		                                                                    BOTH cases implicitly --
		                                                                    the new API splits them)
		Old @dlt.table / @dlt.view code STILL RUNS UNCHANGED -- there is no forced migration.

F8. Declarative dependency resolution (why order-of-writing doesn't matter)

	IMPORTANT mental model shift from procedural pipelines: because Lakeflow Declarative Pipelines
	is declarative, you can literally write the SILVER table's code BEFORE the BRONZE table's code
	in your notebook, and the pipeline engine will still correctly figure out that Bronze must be
	populated FIRST, based on the DATA DEPENDENCY (Silver's query references Bronze), not the
	ORDER the code appears in the file.
	You define WHAT each table's transformation logic is; the engine keeps everything up to date
	as upstream tables change, without you writing explicit "run bronze, then run silver" glue
	code -- this is what makes the pipelines more maintainable and easier to reason about than a
	hand-written procedural DAG of notebook calls.

PART G -- LAKEHOUSE ARCHITECTURE DEEP DIVE (Bronze / Silver / Gold / Data Mesh)
======================================================================================================================================================
======================================================================================================================================================
G1. BRONZE layer -- deep dive

	The Bronze layer is the first, RAWEST layer -- the landing zone for ALL incoming data,
	regardless of format or structure. Its job is simply to preserve ingested data in its
	ORIGINAL state, ensuring it's never lost and is available for later re-processing if needed.

	Key characteristics:
		- UNVALIDATED -- data is not checked for accuracy/completeness; it may contain errors,
		  inconsistencies, or missing values, exactly as it arrived from the source.
		- SCHEMA-ON-READ -- Bronze typically doesn't enforce a rigid schema, allowing flexibility
		  when ingesting from diverse/evolving sources.
		- APPEND-ONLY -- data is appended over time, preserving full history for backtracking/
		  historical trend analysis.
		- LOW COST -- usually implemented on cheap object storage, well-suited to large volumes.

	Should you keep ONE Bronze table per source-type, or separate tables per individual source?
	Depends on:
		1. data volume + ingestion frequency
		2. data lineage/auditability needs
		3. data governance/security requirements
		4. future processing requirements -- if different sources will eventually need DIFFERENT
		   downstream processing logic, separate Bronze tables (and pipelines) per source is
		   cleaner; if the same processing logic applies to all sources regardless of origin, one
		   shared Bronze table can be simpler.
		5. cost -- more tables = more storage/compute management overhead; weigh against the
		   organizational benefit of clearer separation.

	Historical data retention strategy for Bronze -- two common approaches:
		A) ARCHIVE historical data OUT of Bronze into a dedicated archive store, keeping only a
		   limited active window in Bronze.
		   + lower storage cost in Bronze, better query performance (smaller active dataset),
		     simpler governance (active vs archived clearly separated).
		   - needs additional archiving/retrieval infrastructure and process; queries spanning
		     both active+historical data get more complex; lineage/provenance tracking gets
		     harder across two separate stores.
		B) KEEP ALL historical data IN Bronze alongside active data, using time-based
		   partitioning/clustering + solid metadata management for lineage.
		   + simpler (single location for everything), full historical data readily available
		     for analytics/research, complete lineage in one place.
		   - higher storage cost in Bronze over time, can hurt query performance/efficiency at
		     large scale, requires ongoing effort to manage growing volume.
	(In practice: move old data to archive if the business use case is really just "give me the
	latest state" -- keep it all in Bronze if deep historical analysis is a genuine, ongoing need.)
======================================================================================================================================================
G2. SILVER layer -- deep dive

	Silver is the crucial INTERMEDIARY between raw Bronze and curated, analysis-ready Gold. It's
	where raw data gets enriched, validated, and transformed into a more usable shape.

	Key characteristics:
		- STRUCTURED + ENRICHED -- has a defined schema, enabling efficient querying/analysis;
		  involves joining data across sources, deduplication, and data-quality checks.
		- OPTIMIZED FOR CONSUMPTION -- data formatted for specific use cases/analytic tasks; may
		  involve pre-aggregated datasets, materialized views, or calculated metrics for
		  improved query performance.
		- VERSIONED + LINEAGE-TRACKED -- changes are versioned; lineage is documented so you can
		  trace origin and transformations applied.
		- GOVERNED + SECURED -- access-control policies ensure only authorized users see
		  sensitive data; governance practices enforce quality/consistency/reliability.
		- SCALABLE + PERFORMANT -- needs to handle large volumes and concurrent queries
		  efficiently (partitioning/indexing/clustering techniques applied here).

	Transformations that go BEYOND simple cleansing/dedup at the Silver stage:
		- DATA ENRICHMENT -- joining data from multiple sources, calculating derived attributes,
		  adding context-specific information.
		- DATA AGGREGATION -- pre-aggregating for specific use cases to improve downstream query
		  performance.
		- FEATURE ENGINEERING -- creating new features from existing data for ML model input.
		- DATA NORMALIZATION -- converting to a consistent format for easier comparison/analysis.
		- DATA PARTITIONING -- organizing data for efficient querying/analysis.
		- ADVANCED DATA QUALITY CHECKS -- beyond basic validation, things like anomaly detection
		  and completeness analysis.

	Should Silver keep historical data? Depends on: how frequently historical data is actually
	ACCESSED for analysis (frequent access argues for keeping it for fast retrieval), storage
	COST of keeping large historical volumes in Silver, GOVERNANCE requirements mandating certain
	retention periods, and how Silver fits into your broader DATA LIFECYCLE MANAGEMENT strategy.

	Who consumes Silver DIRECTLY (beyond just feeding Gold)?
		- data scientists -- exploratory analysis, model development, feature engineering
		- business analysts -- ad hoc queries, trend/insight analysis
		- data engineers -- monitoring data quality, troubleshooting pipelines, lineage analysis
		- application developers -- accessing pre-processed data for data-driven apps/dashboards

	Rule of thumb: move old data to archive from Silver too, if the actual business use case is
	just "extract the latest data."
======================================================================================================================================================
G3. GOLD layer -- deep dive

	Gold sits at the PINNACLE of the lakehouse -- curated, analysis-ready data drawn from raw
	Bronze and enriched Silver, serving as the SINGLE SOURCE OF TRUTH for business insights and
	decision-making.

	Key characteristics:
		- HIGHLY REFINED -- cleansed, transformed, and aggregated to fit specific business needs
		  and analytic purposes.
		- OPTIMIZED FOR QUERY PERFORMANCE -- data structures/indexes tuned for fast retrieval on
		  complex queries and exploration.
		- BUSINESS-ORIENTED -- organized/presented in a way aligned with how business users
		  actually think about the data, facilitating informed decisions.
		- GOVERNED + SECURED -- strict access controls and governance policies to protect
		  integrity/confidentiality.
		- VERSIONED + LINEAGE-TRACKED -- every change tracked/documented for provenance and
		  consistency over time.

	Benefits: better data quality/consistency for accurate reporting, reduced time-to-insight
	(no more pre-processing needed before analysis), better business decisions from reliable data,
	improved operational efficiency (less time spent on data prep), and a genuine single source of
	truth improving cross-team collaboration/communication.

	Challenges: building/maintaining a robust Gold layer takes real planning/data-modeling/ongoing
	maintenance effort; performance tuning of structures/indexing is an ongoing job; enforcing
	governance/quality standards takes real effort/resources; and there's an inherent tension
	between DATA FRESHNESS and the TIME needed to validate/refine data before it's trustworthy
	enough to serve as Gold.

	Common Gold-layer data models:
		- DIMENSIONAL models -- designed to analyze business processes and relationships between
		  entities (classic star/snowflake schema territory -- facts + dimensions).
		- DATA MARTS -- subject-specific subsets tailored to a particular business area/team.
		- ANALYTICAL DATABASES -- optimized specifically for fast, efficient querying/exploration.

	Best practices: define clear governance policies + access-control rules; establish data
	quality standards + monitoring; automate validation/cleansing processes; thoroughly document
	lineage/transformations; monitor query performance and optimize structures as needed;
	periodically review/update data models/definitions; and communicate clearly with stakeholders
	about data availability/usage guidelines.
======================================================================================================================================================
G4. DATA MESH (your old notes left this section as just a header with no content -- here's the
    full picture, since it's a real, commonly-asked architectural-philosophy topic in interviews)

	Data Mesh is a DECENTRALIZED, organizational approach to data architecture -- proposed by
	Zhamak Dehghani (originally at Thoughtworks) as a response to the pain points large
	organizations hit with a single, centralized data team owning ALL data pipelines for the
	entire company (that central team becomes a bottleneck as the org and its data sources grow).

	Data Mesh rests on FOUR core principles:
		1. DOMAIN-ORIENTED OWNERSHIP -- data is owned and managed by the individual BUSINESS
		   DOMAIN teams that generate/understand it best (e.g. the "orders" team owns order data,
		   the "marketing" team owns campaign data) -- NOT by one central data engineering team
		   that has to become an expert in every domain.
		2. DATA AS A PRODUCT -- each domain team treats the data THEY publish as a first-class
		   PRODUCT with a real "owner," documented quality/SLA guarantees, discoverability, and a
		   defined interface -- the same rigor you'd expect from a proper software API, not "a
		   pile of tables someone happens to maintain."
		3. SELF-SERVE DATA INFRASTRUCTURE AS A PLATFORM -- a central PLATFORM team still exists,
		   but their job shifts to providing reusable, self-serve INFRASTRUCTURE (ingestion
		   tooling, storage, compute, observability) that domain teams use to build/publish their
		   own data products -- rather than the central team building every pipeline itself.
		4. FEDERATED COMPUTATIONAL GOVERNANCE -- governance standards (security, compliance,
		   interoperability, naming conventions) are defined GLOBALLY/collaboratively across
		   domains, but ENFORCED in a federated, largely automated way (via the shared platform)
		   rather than requiring every change to pass through one central gatekeeping team.

	Data Mesh vs Medallion/Lakehouse architecture -- these are NOT competing alternatives, they
	answer DIFFERENT questions:
		- Medallion (Bronze/Silver/Gold, Parts G1-G3) is about the TECHNICAL/PHYSICAL LAYERING of
		  data as it gets progressively refined -- a pattern for HOW data gets transformed.
		- Data Mesh is about ORGANIZATIONAL OWNERSHIP -- WHO is responsible for which data, and
		  how teams collaborate/govern it at scale.
		- In practice, a large organization can absolutely run a Data Mesh operating model WHERE
		  each domain's own data lives in ITS OWN medallion (Bronze/Silver/Gold) pipeline, on a
		  SHARED Databricks + Unity Catalog platform. Domain teams own their own Bronze->Silver->
		  Gold logic for their data; the platform team provides the shared Databricks/Unity
		  Catalog/Lakeflow infrastructure everyone builds on.

	How Databricks maps onto Data Mesh in 2026:
		- Unity Catalog's 3-level namespace (catalog.schema.table, E3) naturally supports
		  domain-based catalogs (e.g. one catalog per business domain).
		- Unity Catalog DOMAINS (E10, new in 2026) directly support grouping data/AI assets by
		  business domain -- a very literal platform-level implementation of "domain ownership."
		- Delta Sharing / OpenSharing (E11) is the natural mechanism for one domain to PUBLISH its
		  "data product" for other domains to consume, without giving them direct write access or
		  requiring a data copy.
		- Unity Catalog governance (ABAC, lineage, auditing -- Part E) is the FEDERATED
		  COMPUTATIONAL GOVERNANCE layer -- centrally defined policy, automatically enforced
		  wherever the data lives/travels.
		- Unity Catalog Metrics + Business Glossary (E9-E10) help each domain formally define what
		  their "data product" actually means/contains, which is core to the "data as a product"
		  principle.

H1. LAKEBASE -- fully managed, serverless Postgres OLTP database

	Historically, Databricks had NO real operational/transactional (OLTP) database story -- if
	your APPLICATION needed a normal request/response database (the kind that backs a web app's
	"create an order," "update a user profile" actions), you'd run a completely separate Postgres/
	MySQL/etc. service OUTSIDE Databricks, and build a custom CDC pipeline to get that data INTO
	the lakehouse for analytics. Lakebase closes that gap.

	Lakebase is a fully managed Postgres database, BUILT INTO Databricks, using technology from
	Neon (a company Databricks acquired) as its foundation.

	Why it behaves differently from a normal, monolithic Postgres install: Postgres COMPUTE is
	made STATELESS by externalizing the write-ahead log (WAL) and data files into independent,
	distributed storage services (a "SafeKeeper" service for the WAL, a "PageServer" service for
	the actual data pages). Because compute no longer OWNS the data on local disk, you get
	capabilities a traditional Postgres install structurally can't offer:
		- INSTANT, GIT-STYLE BRANCHING -- branch a copy of your production database for dev/test
		  in seconds (copy-on-write, not a physical data copy), and reset it to prod state anytime.
		- SCALE-TO-ZERO -- compute suspends automatically when idle; you're not paying for a
		  database instance sitting around unused.
		- POINT-IN-TIME RECOVERY + cross-cloud/cross-region DISASTER RECOVERY.
		- Up to ~5x FASTER WRITE THROUGHPUT on write-heavy OLTP workloads -- achieved by offloading
		  Postgres's expensive "Full Page Write" crash-recovery mechanism from compute onto the
		  distributed storage layer (storage reconstructs pages from image + WAL-delta chains
		  instead of compute having to write whole 8KB page images into the WAL on every first-
		  change-after-checkpoint, which is what makes stock Postgres WAL volume balloon under
		  heavy write load).

	Practical USE CASES:
		- backend database for Databricks Apps (Part H4)
		- an ONLINE FEATURE STORE for ML models (low-latency feature lookups at inference time)
		- a STATE/SESSION STORE for AI agents (so an agent can "remember" a conversation across
		  multiple turns, or even across separate deployments/restarts)
		- syncing Unity Catalog (lakehouse) tables INTO Lakebase for low-latency application reads
		  (an app reading from Lakebase gets response-time performance a Spark-based lakehouse
		  query engine isn't built to deliver for single-row lookups)

	LAKEBASE CHANGE DATA FEED -- captures changes made IN Postgres and exposes them AS Delta
	tables automatically, for downstream analytics -- this is the "reverse direction" of the old
	problem statement (getting APP data into the LAKEHOUSE), and it requires no custom CDC
	pipeline you build/maintain yourself.
======================================================================================================================================================
H2. LTAP -- Lake Transactional/Analytical Processing (announced Summit 2026, "coming soon")

	LTAP is a new ARCHITECTURE (not just a product) that aims to unify OLTP (Lakebase/Postgres)
	and OLAP (the Lakehouse) on ONE copy of data, stored in open formats, governed by a SINGLE
	Unity Catalog model.

	The GOAL: a transaction WRITTEN to Lakebase should be IMMEDIATELY QUERYABLE in the Lakehouse
	-- eliminating the CDC pipeline, the second copy of the data, and the ETL step that
	traditionally sits between "the app's database" and "the analytics warehouse."
	Under the hood: transactional workloads still run as REAL Postgres with full ACID semantics;
	analytical workloads still run across the full Lakehouse at scale; each SCALES INDEPENDENTLY
	(so a huge analytical query doesn't slow down the app's transactional writes, and vice versa).
	Because LTAP is built on open standards (Postgres wire protocol + Iceberg/Delta table
	formats), it works with any application that already speaks Postgres, and any reader that
	already understands open table formats.
	Powered by a new query engine Databricks calls REYDEN (also powers Lakehouse//RT -- H3).

	INTERVIEW-GRADE NUANCE (worth having ready if asked to go deeper): independent technical
	analysis published after the Summit 2026 announcement (The Register, July 2026) points out
	that in practice there ARE two distinct physical representations of the data kept
	continuously in sync -- Postgres's own "pageserver" format used for serving transactional
	reads/writes, and a columnar/Parquet representation used for analytics. So Databricks'
	marketing claim of "zero copies" is more accurately understood as "zero MANUAL/ETL copies" --
	you never write or maintain a CDC/copy pipeline yourself -- rather than literally one single
	physical byte-for-byte copy sitting on disk. This is a genuinely good nuance to volunteer in
	an interview if the topic comes up, since it shows you dug past the marketing headline.
======================================================================================================================================================
H3. Lakehouse//RT -- real-time SQL warehouse (Beta)

	A NEW warehouse TYPE (distinct from the classic/serverless DBSQL warehouses you already know),
	also powered by the Reyden query engine, purpose-built for SUB-SECOND, HIGH-CONCURRENCY BI
	workloads -- the kind of "hundreds of dashboard users hitting refresh at once, and it needs
	to feel instant" scenario that a traditional batch-oriented data warehouse struggles with.
	Databricks-reported benchmark: roughly 10x faster query response on some real-time benchmarks
	vs. the prior-generation warehouse type, in early customer testing (e.g. a healthcare
	customer case study cited in Databricks' own materials).
======================================================================================================================================================
H4. Databricks Apps

	Lets you build and deploy small, INTERACTIVE WEB APPLICATIONS (think Streamlit/Gradio/Flask-
	style apps) directly INSIDE your Databricks workspace -- backed by Lakebase (H1) for any
	transactional/state needs, and governed by Unity Catalog for data access -- WITHOUT having to
	stand up separate hosting infrastructure just to give a business team a lightweight internal
	tool (e.g. "a simple form for the ops team to approve/reject flagged transactions," backed
	directly by your governed lakehouse data).
	Extensions announced at Summit 2026: App Spaces, and (AI-flavored) Genie App Builder /
	Serverless Micro Apps for turning a Genie space or an agent into a lightweight app -- see the

J3. Executor memory internals -- Unified Memory Management

	Within EACH executor's on-heap memory (Part A8), Spark splits usable memory into two
	dynamically-shared regions:
		- EXECUTION memory -- used for shuffles, joins, sorts, aggregations (the actual compute)
		- STORAGE memory -- used for caching/persisting DataFrames (Part C5) and broadcast variables
	"Unified" because these two regions can BORROW from each other as needed (execution can evict
	cached storage data under memory pressure, and vice versa within limits) rather than each
	having a hard, wasteful, fixed reservation.
		spark.memory.fraction        -- fraction of JVM heap used for this unified region
		                                 (rest is reserved for Spark's own internal metadata/objects)
		spark.memory.storageFraction -- fraction of the unified region protected as a MINIMUM for
		                                 storage (cached data) before execution is allowed to evict it
	Interview framing: "why doesn't a big .cache() call always cause an OOM error?" -- because
	Spark will EVICT the least-recently-used cached partitions to make room for execution memory
	instead of crashing, as long as execution memory alone can fit.

J4. Cluster sizing -- a worked example (classic interview question at every big tech company)

	Q: "You have a cluster of 5 worker nodes, each with 16 cores and 64GB RAM. How would you
	configure executors?"

	Reasoning steps interviewers want to see:
		1. Reserve ~1 core + ~1GB per node for the OS/daemons/overhead.
		2. Rule of thumb: cap executor cores at ~5 (more than that per executor tends to hurt HDFS/
		   cloud-storage I/O throughput and increases GC pause impact if one executor holds too
		   many concurrent tasks) -- the same "5 cores per executor" number from Part A5.
		3. With 15 usable cores per node / 5 cores per executor = 3 executors per node.
		4. 5 nodes x 3 executors = 15 executors total, minus 1 reserved for the driver = 14
		   executors available for the application (adjust for client vs cluster deploy mode, J2).
		5. Memory per executor = (usable RAM per node) / (executors per node), minus ~7% overhead
		   Spark reserves per executor for off-heap/JVM overhead (spark.executor.memoryOverhead).
	The exact "right" answer varies (deliberately a discussion question, not one correct number) --
	what matters is demonstrating you know the TRADE-OFFS: more executors with fewer cores each =
	better I/O parallelism & fault isolation but more per-executor JVM overhead; fewer executors
	with more cores each = less overhead but worse I/O parallelism and a bigger blast radius if one
	executor dies.
	On Databricks specifically, serverless compute (Part A9) and autoscaling take this manual
	decision away from you for most workloads -- but the underlying trade-offs are exactly what
	autoscaling/serverless optimize on your behalf, and interviewers ask this to test whether you
	understand WHY, not just whether you can recite a formula.

J5. Classic RDD interview comparison: reduceByKey vs groupByKey -- and accumulators (a genuine gap)

	groupByKey()   -- shuffles ALL values for a key across the network FIRST, THEN groups them --
	                  expensive, can cause massive shuffle volume/OOM on skewed/large data.
	reduceByKey()  -- combines values LOCALLY on each partition FIRST (a "map-side combine",
	                  similar in spirit to a MapReduce Combiner), and only shuffles the much
	                  smaller PARTIALLY-REDUCED results -- far less network I/O.
	Rule of thumb (asked constantly): prefer reduceByKey/aggregateByKey over groupByKey whenever
	computing an aggregate, since groupByKey shuffles raw, un-aggregated data while reduceByKey
	shuffles pre-combined, smaller data.
	2026 practical note: on the DataFrame/SQL API (what you actually use day to day, Part A6),
	Catalyst already picks the efficient equivalent of "reduceByKey-style" partial aggregation
	automatically for groupBy().agg(...) -- this comparison mostly tests whether you understand
	the underlying shuffle-cost concept, not something you manually choose between in modern code.

	ACCUMULATORS (a DIFFERENT concept from broadcast variables, Part C7 -- commonly confused):
	a broadcast variable sends a READ-ONLY copy of data OUT to every executor; an accumulator is
	the OPPOSITE direction -- a WRITE-ONLY (from the executors' perspective) variable that
	executors can only ADD TO, which the DRIVER then reads the final aggregated total of. Classic
	use: counting how many malformed/bad records were skipped across an entire distributed job.
		bad_record_count = spark.sparkContext.accumulator(0)
		def process(row):
		    if is_bad(row):
		        bad_record_count.add(1)
		df.foreach(process)
		print(bad_record_count.value)   -- read only on the driver, after an action has run
	Caution (a real interview gotcha): accumulator updates inside TRANSFORMATIONS (not actions)
	can be applied MORE THAN ONCE if a stage gets recomputed (e.g. a lost partition) --
	accumulators are only guaranteed exactly-once for updates made inside ACTIONS.

J6. SQL/DataFrame join types (a genuine gap -- your old notes covered join MECHANICS/internals,
    Sort-Merge, broadcast, but never the actual join TYPES themselves)

	INNER JOIN         -- only matching rows from both sides
	LEFT (OUTER) JOIN   -- all rows from the left side, matched columns from the right (NULL if no match)
	RIGHT (OUTER) JOIN  -- all rows from the right side, matched columns from the left (NULL if no match)
	FULL (OUTER) JOIN   -- all rows from both sides, NULL where there's no match on either side
	LEFT SEMI JOIN      -- returns rows from the LEFT side that HAVE a match on the right, but
	                        includes ONLY left-side columns (like "filter using a lookup table",
	                        NOT an actual join output with right-side columns) -- commonly confused
	                        with a plain inner join + select
	LEFT ANTI JOIN      -- returns rows from the LEFT side that have NO match on the right --
	                        the "find records missing from the other table" pattern (an alternative
	                        to subtract()/exceptAll from Part B17, expressed as a join instead)
	CROSS JOIN          -- Cartesian product, every row from the left paired with every row from
	                        the right -- rarely intentional; a well-known PERFORMANCE ANTI-PATTERN
	                        when it happens BY ACCIDENT (e.g. forgetting a join condition)

	df1.join(df2, "id", "inner")
	df1.join(df2, "id", "left")
	df1.join(df2, "id", "left_semi")
	df1.join(df2, "id", "left_anti")
	df1.crossJoin(df2)

J7. Join execution strategies (beyond Broadcast and Sort-Merge, Parts C7/C9)

	Spark actually has FOUR join strategies it can pick between (chosen by the Catalyst/AQE cost
	model, Part C6):
		1. BROADCAST HASH JOIN -- C7. Used when one side is small enough to broadcast.
		2. SHUFFLE HASH JOIN -- shuffles both sides by join key (like Sort-Merge), but builds an
		   in-memory HASH TABLE on the smaller side per-partition instead of sorting both sides.
		   Cheaper than Sort-Merge when one side, post-shuffle, is small enough to fit comfortably
		   in memory per partition -- but riskier (can OOM) if that assumption is wrong, which is
		   why Spark generally prefers Sort-Merge as the "safe default" for two large sides.
		3. SORT-MERGE JOIN -- C9. The default, safe choice for two large sides -- doesn't require
		   either side to fit in memory, since it processes sorted streams incrementally.
		4. BROADCAST NESTED LOOP JOIN -- the FALLBACK when there's no equality join condition to
		   hash/sort on at all (e.g. a join condition using "<", ">", or a non-equality predicate)
		   -- compares every row of one side against every row of the other, broadcasting the
		   smaller side. Very expensive; a common cause of an unexpectedly slow query is an
		   accidental non-equi-join condition forcing this strategy.
	Interview framing: "why would Spark choose Sort-Merge over Shuffle Hash even when Shuffle Hash
	is theoretically cheaper?" -- answer: safety. Sort-Merge doesn't risk an OOM if the size
	estimate is wrong; Shuffle Hash does. AQE's runtime re-planning (Part C6) reduces how often
	that estimate is wrong in the first place.

J9. Common performance anti-patterns ("a job is slow, what do you check first?" -- a very
    common scenario-based interview question)

	1. df.collect() on a large DataFrame -- pulls ALL data to the driver's memory; the classic
	   cause of a driver OOM. Use .take(n)/.show(n)/.limit(n) for inspection instead.
	2. Calling .count() (or other actions) repeatedly on the SAME unmaterialized DataFrame inside
	   a loop -- each call re-executes the ENTIRE lineage from scratch -- cache/persist (Part C5)
	   first if you're going to hit the same DataFrame with multiple actions.
	3. Accidental CROSS JOIN (J6) from a missing/wrong join condition -- always sanity-check row
	   counts after a join against expectations.
	4. Chaining MANY .withColumn() calls in a loop instead of one .select() (Part B14).
	5. Using a Python/Pandas UDF where a built-in function exists (Parts B8/C12) -- loses Catalyst
	   optimization AND forces a Photon fallback.
	6. Ignoring the "small file problem" (Parts C1/D13) -- thousands of tiny files from
	   over-partitioned writes or high-frequency streaming micro-batches without periodic compaction.
	7. Not filtering EARLY -- relying entirely on Catalyst's predicate pushdown instead of writing
	   filters as close to the source read as possible when reading from non-Delta sources that
	   don't support pushdown as well.
	8. Over-partitioning shuffle output for a small dataset (e.g. leaving shuffle partitions at
	   200 for a dataset that easily fits in 4 partitions) -- creates unnecessary task-scheduling
	   overhead (mostly mitigated by AQE auto-coalescing now, Part C6, but still worth knowing as
	   a root cause on older Spark versions).

J10. repartitionByRange

	df.repartitionByRange(10, "event_date")
	Unlike repartition() (Part C2), which hash-partitions data (a given key always lands in the
	same partition, but partitions aren't internally ORDERED relative to each other),
	repartitionByRange creates partitions based on a RANGE of the sort key's values (e.g.
	partition 1 = Jan-Mar, partition 2 = Apr-Jun...) using sampling to estimate good range
	boundaries. Useful when downstream processing benefits from ROUGHLY sorted, range-based
	partitions (e.g. writing time-series data out in date order) rather than just even distribution.

J11. Window function frame specifications + ranking function nuances

	Beyond the basic Window.partitionBy().orderBy() from Part B18, you can control exactly which
	ROWS are included in each window's calculation:

	ROWS BETWEEN <n> PRECEDING AND CURRENT ROW    -- physical row count, e.g. a 7-row rolling window
	RANGE BETWEEN <n> PRECEDING AND CURRENT ROW   -- value-based range on the ORDER BY column
	                                                   (e.g. all rows within the last 7 DAYS of the
	                                                   current row's date, regardless of how many
	                                                   physical rows that is)
	ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW  -- classic running total pattern

	w = Window.partitionBy("customer_id").orderBy("order_date") \
	    .rowsBetween(Window.unboundedPreceding, Window.currentRow)
	df.withColumn("running_total", sum("amount").over(w))

	RANKING functions -- commonly confused with each other in interviews:
		row_number()    -- unique, sequential rank, NO ties (1,2,3,4,5 always, even with equal values)
		rank()          -- ties get the SAME rank, but the NEXT rank SKIPS accordingly (1,2,2,4,5)
		dense_rank()    -- ties get the SAME rank, next rank does NOT skip (1,2,2,3,4)
		percent_rank()  -- relative rank as a percentage (0.0 to 1.0) of the window
		ntile(n)        -- splits the window into n roughly-equal buckets (e.g. ntile(4) = quartiles)

J12. PIVOT / UNPIVOT and CTEs in Spark SQL

	PIVOT -- turn distinct row values into columns (classic "rows to columns" reshape):
		SELECT * FROM (SELECT dept, month, sales FROM sales_data)
		PIVOT (SUM(sales) FOR month IN ('Jan','Feb','Mar'));

	UNPIVOT (Spark SQL, newer syntax support) -- the reverse: columns back into rows.

	CTE (Common Table Expression, the WITH clause) -- named, reusable sub-query within a single
	statement, improves readability over deeply nested sub-selects:
		WITH recent_orders AS (
		    SELECT * FROM orders WHERE order_date > current_date() - INTERVAL 30 DAYS
		)
		SELECT customer_id, COUNT(*) FROM recent_orders GROUP BY customer_id;
	Spark 4.0's new SQL SCRIPTING support (Part A10) extends this further with session variables
	and multi-statement procedural SQL blocks, not just single-statement CTEs.

J13. Delta Lake Change Data Feed (CDF) -- a genuine gap versus your old notes

	CDF is DIFFERENT from Lakebase's "Change Data Feed" (Part H1, which is about Postgres->Delta
	sync) -- this is a Delta TABLE-level feature for capturing ROW-LEVEL CHANGES (inserts/updates/
	deletes) made to any Delta table, so DOWNSTREAM consumers can process just the CHANGES since
	they last read, instead of re-reading/diffing the entire table.

	Enable it (settable at creation, or altered on later):
		ALTER TABLE my_table SET TBLPROPERTIES (delta.enableChangeDataFeed = true);

	Read the change feed (as a batch range, or as a stream):
		spark.read.format("delta") \
		    .option("readChangeFeed", "true") \
		    .option("startingVersion", 5) \
		    .table("my_table")

	The result includes normal columns PLUS three metadata columns:
		_change_type      -- 'insert', 'update_preimage', 'update_postimage', or 'delete'
		_commit_version   -- which table version this change came from
		_commit_timestamp -- when that commit happened

	WHY IT MATTERS: this is the standard mechanism for propagating changes DOWNSTREAM through a
	medallion architecture (Part G) efficiently -- e.g. Silver only needs to re-process rows that
	actually CHANGED in Bronze, instead of a full re-scan every run. It's also commonly used as
	the source for SCD Type 2 (Part D10) implementations and for feeding incremental updates into
	a Lakebase sync (Part H1) or an AI Vector Search index refresh (AI notepad Part B5).

J14. Delta table constraints & special column types

	NOT NULL constraint:
		ALTER TABLE my_table CHANGE COLUMN customer_id SET NOT NULL;

	CHECK constraint (arbitrary boolean expression enforced on every write):
		ALTER TABLE my_table ADD CONSTRAINT valid_amount CHECK (amount >= 0);

	PRIMARY KEY / FOREIGN KEY -- Delta supports declaring these, but IMPORTANT interview nuance:
	they are INFORMATIONAL ONLY by default (NOT enforced like in a traditional RDBMS) -- they
	exist mainly to document intent and help query optimizers/BI tools understand relationships,
	NOT to reject a bad insert automatically. (Databricks has been extending enforcement options
	over time -- always verify current enforcement behavior against docs before relying on it as a
	data-quality guardrail; treat NOT NULL/CHECK as the constraints you can actually depend on to
	REJECT bad data.)

	GENERATED COLUMNS -- a column whose value is automatically computed from other columns on
	every write, rather than supplied by the writer:
		CREATE TABLE events (
		  event_time TIMESTAMP,
		  event_date DATE GENERATED ALWAYS AS (CAST(event_time AS DATE))
		) USING DELTA;
	Commonly used to derive a partition-friendly/clustering-friendly column (like a date) from a
	finer-grained timestamp column automatically.

	IDENTITY COLUMNS -- auto-incrementing surrogate key columns, similar to IDENTITY/SERIAL in a
	traditional RDBMS:
		CREATE TABLE dim_customer (
		  customer_key BIGINT GENERATED ALWAYS AS IDENTITY,
		  customer_name STRING
		) USING DELTA;
	Useful for generating surrogate keys in dimension tables (SCD1/SCD2, Parts D8/D10) without a
	separate sequence-generation mechanism.

J16. Structured Streaming: watermarking & handling late-arriving data

	A genuine gap versus your old notes (Parts B1/F1 mentioned streaming basics and its
	LIMITATIONS -- no sorting, no full dedup -- but not the actual mechanism for handling late
	data at all).

	PROBLEM: in a streaming aggregation (e.g. "count events per 10-minute window"), what do you do
	about an event that arrives LATE -- after you've already emitted a result for its window?
	Keeping ALL windows open forever to handle arbitrarily-late data would grow STATE unboundedly
	and never let Spark free old, no-longer-needed state.

	WATERMARKING is the answer -- you tell Spark how late you're willing to tolerate data being:
		df.withWatermark("event_time", "10 minutes") \
		  .groupBy(window("event_time", "5 minutes")) \
		  .count()
	This tells Spark: "once you've seen an event_time value, assume no more data will arrive more
	than 10 minutes older than the MAX event_time seen so far -- once that threshold passes for a
	given window, finalize its result and DROP its state." Data arriving even later than the
	watermark allows is silently dropped from that aggregation.
	This is ALSO the mechanism that finally makes DEDUPLICATION possible on a streaming
	DataFrame (your old notes correctly note plain streaming can't dedupe globally) --
	dropDuplicatesWithinWatermark() dedupes within the bounded watermark window instead of
	requiring unbounded state to check the entire history of the stream.

J17. Structured Streaming joins

	STREAM-STATIC JOIN -- joining a streaming DataFrame against a (much smaller, relatively
	stable) STATIC/batch DataFrame -- e.g. joining a stream of orders against a static dimension
	table of customers. No watermark needed on the static side.
		stream_df.join(static_dim_df, "customer_id")

	STREAM-STREAM JOIN -- joining TWO streaming DataFrames against each other (e.g. joining a
	stream of ad impressions against a stream of ad clicks to compute click-through attribution).
	REQUIRES a watermark on BOTH sides (and typically a time-range join condition), since Spark
	needs to know how long to keep buffering unmatched rows from each side waiting for a
	late-arriving match on the other side, or state would grow unboundedly.

J18. Stateful streaming operations (conceptual)

	Beyond simple windowed aggregations (J16), Spark supports fully CUSTOM stateful processing via
	mapGroupsWithState / flatMapGroupsWithState -- you write your own function that receives the
	new micro-batch of data for a key PLUS whatever state you've chosen to keep from previous
	batches for that key, and returns an updated state plus optionally some output. Used for
	custom scenarios simple windowing can't express (e.g. session-based analytics: "group events
	into a user session that ends after 30 minutes of inactivity," which isn't a fixed time
	window). This is an advanced/niche topic -- know the NAME and the CONCEPT for interviews; most
	real pipelines use the simpler windowed-aggregation pattern from J16 instead.

J20. Cluster access modes & Unity Catalog compatibility (an admin-adjacent topic that shows up

	Relevant for interview questions about "how would you automate deploying/managing Databricks

J24. Row-level security & column masking

	LEGACY approach (pre/early Unity Catalog) -- DYNAMIC VIEWS using CASE WHEN + is_member():
		CREATE VIEW secure_salary_view AS
		SELECT
		  employee_id,
		  CASE WHEN is_member('hr_group') THEN salary ELSE NULL END AS salary
		FROM employees;
	Every "row-level security" or "column masking" implementation before Unity Catalog's native
	support was essentially a hand-written view like this.

	MODERN (Unity Catalog native, ties to ABAC -- Part E7) -- ROW FILTERS and COLUMN MASKS are
	first-class Unity Catalog objects you attach directly to a table, rather than hand-writing a
	wrapping view:
		CREATE FUNCTION mask_salary(salary DOUBLE) RETURNS DOUBLE
		  RETURN CASE WHEN is_member('hr_group') THEN salary ELSE NULL END;
		ALTER TABLE employees ALTER COLUMN salary SET MASK mask_salary;
	Benefit over the legacy view-based approach: the masking rule now lives ON the table itself
	(governed/auditable as a Unity Catalog object) and applies EVERYWHERE the table is queried
	from -- including via Catalog Federation from an external engine (Part E8) -- rather than only
	protecting people who happen to query through your specific hand-built view.

J26. Testing PySpark code (a real gap -- genuinely common in senior data-engineer interviews:
     "how do you unit test a Spark transformation?")

	Treat a PySpark transformation function like any other testable code -- write it as a PURE
	function that takes a DataFrame in and returns a DataFrame out, so it's testable without a
	full production-scale dataset:

		def clean_orders(df):
		    return df.filter(col("amount") > 0).dropDuplicates(["order_id"])

		# test file, using pytest:
		def test_clean_orders(spark_session):
		    input_df = spark_session.createDataFrame(
		        [(1, 10.0), (1, 10.0), (2, -5.0)], ["order_id", "amount"]

		    assert result.count() == 1

	Common tooling: pytest (test runner) + a small local SparkSession fixture (a single-node local
	Spark instance, fast to spin up for tests) + a comparison helper library like chispa (or
	spark-testing-base) for readable DataFrame-equality assertions instead of hand-rolling
	row-by-row comparison logic.

J29. SQL coding-interview patterns worth having ready (adjacent to, but distinct from,
     Databricks-specific knowledge -- these show up constantly in data-engineering/analytics
     interviews at every big tech company, regardless of platform)

	SECOND HIGHEST VALUE (classic "Nth highest salary" pattern):
		SELECT MAX(salary) FROM employees
		WHERE salary < (SELECT MAX(salary) FROM employees);
		-- or, more flexibly for "Nth highest": DENSE_RANK() OVER (ORDER BY salary DESC) = N

	RUNNING TOTAL:
		SELECT order_date, amount,
		  SUM(amount) OVER (ORDER BY order_date ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW)
		  AS running_total
		FROM orders;

	GAPS AND ISLANDS (find consecutive runs / find missing values in a sequence) -- classic
	pattern: subtract a ROW_NUMBER() from a date/sequence column; consecutive rows in the same
	"island" end up with the SAME resulting value (since both the sequence and the row number
	increment together for consecutive rows), letting you GROUP BY that difference to identify
	each island:
		SELECT *, DATE_SUB(event_date, ROW_NUMBER() OVER (ORDER BY event_date)) AS island_group
		FROM events;
	These three patterns (Nth value, running total, gaps-and-islands) cover a large fraction of
	"write a SQL query to..." interview questions across companies -- worth having the shape of
	each memorized cold, since the underlying window-function toolkit (Parts B18/J11) is exactly
	what you already know from the Databricks side.

K1. Tungsten execution engine & Whole-Stage Code Generation (predates Photon -- a real gap, since
    Part C12 explained Photon without contrasting it against what Spark SQL already did before)

	TUNGSTEN (built into stock/open-source Spark since 2015, still the engine running under
	Photon's hood when Photon ISN'T handling an operator) improved Spark SQL performance via:
		- OFF-HEAP, BINARY memory management for DataFrame rows (avoids JVM object overhead/GC
		  pressure per-row, unlike raw RDDs of Java/Python objects)
		- CACHE-AWARE computation, structured to be friendly to modern CPU cache hierarchies
		- WHOLE-STAGE CODE GENERATION -- instead of interpreting a query plan operator-by-operator
		  at runtime (calling a generic "evaluate this expression" function repeatedly, with
		  virtual-function-call overhead every time), Spark GENERATES actual Java bytecode
		  SPECIFIC to your query, fusing multiple operators (filter + project + aggregate, say)
		  into ONE tight compiled loop -- much closer to what a hand-written, hard-coded Java
		  program doing that exact same operation would look like, instead of a slow, generic,
		  interpreted execution path.
	df.explain("codegen")   -- shows the actual generated Java code for a query (rarely read line-
	                            by-line in practice, but knowing this command exists, and WHY it
	                            exists, is what interviewers are actually checking for)

	TUNGSTEN vs PHOTON -- the relationship, precisely:
		- Tungsten is part of STOCK Apache Spark (JVM-based, row-oriented internally at the
		  whole-stage-codegen level, though it also has some vectorized readers).
		- Photon (Part C12) is Databricks' PROPRIETARY, NATIVE C++ engine that REPLACES Tungsten's
		  execution for the operators it supports, processing data in true COLUMNAR BATCHES
		  (thousands of rows at once) rather than Tungsten's row-at-a-time (even if fused/compiled)
		  model -- which is WHY Photon is faster still: it's not just "compiled code" (Tungsten
		  already does that), it's compiled, VECTORIZED, columnar code with no JVM/GC involvement
		  for the operators it covers at all.
		- When Photon falls back (Part C12 -- UDFs etc.), it falls back to TUNGSTEN-powered plain
		  JVM Spark, not to some even-more-primitive interpreter -- so a "Photon fallback" still
		  benefits from whole-stage codegen, just not from Photon's columnar/native execution.
	Interview-ready one-liner: "Tungsten made Spark SQL fast by compiling queries to bytecode and
	managing memory off-heap in binary form; Photon goes further by re-implementing execution
	natively in C++ with true columnar/vectorized processing, for the operators it supports."

K2. Serialization: Kryo vs Java (a classic Spark tuning question, not previously covered)

	Whenever Spark needs to move data across the network (shuffles) or spill it to disk, objects
	must be SERIALIZED (converted to bytes) and later DESERIALIZED. The serializer you use
	materially affects shuffle performance:

		spark.conf.set("spark.serializer", "org.apache.spark.serializer.KryoSerializer")

		Java serialization (DEFAULT if unset) -- built into the JVM, works for any Serializable
		  object with zero extra setup, but SLOW and produces LARGE serialized output.
		Kryo serialization -- a third-party library Spark integrates with, meaningfully FASTER and
		  produces SMALLER serialized output than Java serialization -- the well-known "just turn
		  this on" performance win for RDD-heavy or custom-object-heavy workloads. Requires
		  registering custom classes for best performance (Kryo can still work without explicit
		  registration, just slightly less optimally).
	2026 practical note: this matters far less for standard DataFrame/SQL API workloads (Part A6)
	since Tungsten (K1) already uses its own optimized binary row format for DataFrames rather
	than falling back to generic Java/Kryo object serialization -- Kryo-vs-Java is mostly an RDD-
	API-era/custom-object tuning question now, but it's still asked to test foundational understanding.

K3. Speculative execution & Dynamic Resource Allocation

	SPECULATIVE EXECUTION -- if one task within a stage is running MUCH slower than its sibling
	tasks (a "straggler," often caused by a flaky/overloaded node rather than actual data skew),
	Spark can proactively launch a DUPLICATE copy of that same task on a different executor, and
	simply uses WHICHEVER copy finishes first, killing the other.
		spark.conf.set("spark.speculation", True)
	Distinct from AQE's skew handling (Part C6) -- skew handling addresses UNEVEN DATA causing a
	task to have more genuine work to do; speculative execution addresses a task that's slow for
	INFRASTRUCTURE reasons (a slow/degraded node) despite having a normal amount of work.

	DYNAMIC RESOURCE ALLOCATION -- the underlying (open-source) Spark mechanism that lets an
	application's executor COUNT grow/shrink WHILE IT'S RUNNING, based on the current backlog of
	pending tasks, instead of holding a fixed number of executors for the application's entire
	lifetime.
		spark.conf.set("spark.dynamicAllocation.enabled", True)
	This is the underlying Spark-level mechanism that Databricks' CLUSTER AUTOSCALING (Part A9)
	and SERVERLESS compute's automatic scaling both build on/abstract away for you -- worth
	knowing the plain-Spark primitive underneath the Databricks-managed convenience feature,
	since interviewers who come from a general Spark background may ask about it by this name.

K4. Delta Lake concurrency control (a real gap -- "how does Delta handle two writers at once?"
    is a very commonly asked question this handbook hadn't answered directly)

	Delta Lake uses OPTIMISTIC CONCURRENCY CONTROL for writes -- it does NOT lock the table while
	a write is in progress. Instead:
		1. a writer reads the table's current version, computes its changes
		2. at COMMIT time, it checks whether any OTHER writer has committed a conflicting change
		   to the SAME files/version since it started
		3. if there's no conflict, its commit (a new JSON file in _delta_log, Part D2) succeeds
		4. if there IS a conflict, the SECOND writer's transaction FAILS (throwing a specific
		   conflict exception) rather than silently corrupting data -- that writer is expected to
		   RETRY (re-read the new current state and re-attempt its change)
	This is "optimistic" specifically because it assumes conflicts are RARE and only pays the cost
	of detecting/handling one when it actually happens, rather than pessimistically locking on
	every write (which would hurt throughput for the common, non-conflicting case).

	Common conflict exceptions you may see (useful to recognize by name in an interview):
		ConcurrentAppendException       -- two writers both tried to append/modify files that
		                                     overlap in a way that can't be reconciled
		ConcurrentDeleteReadException    -- one writer deleted files that another writer's
		                                     transaction had already read and depended on
		MetadataChangedException         -- the table's schema/metadata changed underneath a
		                                     writer's in-flight transaction

	ISOLATION LEVELS -- Delta Lake supports:
		WriteSerializable (DEFAULT) -- slightly relaxed compared to full Serializable, allows
		  more concurrent writes to succeed (better throughput) while still preventing the most
		  problematic anomalies -- the right default for most workloads.
		Serializable -- the strictest level, guarantees the outcome is equivalent to the writes
		  having happened in SOME strict sequential order -- can be set per-table if your use case
		  genuinely needs that stronger guarantee, at the cost of more transactions failing/retrying
		  under concurrent load.
	2026 note: Delta Lake 4.x's "coordinated commits" (Part D24) extends this same optimistic-
	concurrency philosophy to allow MULTIPLE DIFFERENT ENGINES (not just multiple Spark writers)
	to safely participate in the same commit-conflict-detection protocol on one table.

K5. Table statistics, ANALYZE TABLE, and cost-based optimization (CBO)

	Catalyst's cost-based decisions (e.g. "is this table small enough to broadcast," Part C7) rely
	on STATISTICS about the table -- row counts, column min/max/null-counts/distinct-counts.
	Delta automatically captures SOME of this (the min/max stats used for data skipping, Part D17)
	as part of normal writes, but for the OPTIMIZER to make good cost-based decisions (not just
	data-skipping), more complete statistics sometimes need to be computed explicitly:

		ANALYZE TABLE my_table COMPUTE STATISTICS;
		ANALYZE TABLE my_table COMPUTE STATISTICS FOR COLUMNS col1, col2;

	Why this still matters even with AQE (Part C6): AQE improves decisions made AFTER a stage has
	already run (using ACTUAL runtime sizes), but the INITIAL plan Catalyst starts with is still
	built from stats-based estimates -- better upfront statistics mean a better STARTING plan, so
	AQE has less work to correct at runtime, and some decisions (like initial join ordering across
	more than 2 tables) are made before AQE gets a chance to intervene at all.

K6. Auto Loader schema evolution modes + the rescued data column (Auto Loader-specific gap --
    Part F5 covered Auto Loader's file-detection mechanics but not its schema-handling behavior)

	cloudFiles.schemaLocation -- REQUIRED when you want Auto Loader to track/infer schema over
	  time (a location where it persists the schema it has inferred so far, across restarts).

	cloudFiles.inferColumnTypes -- whether to infer actual DATA TYPES (int, double, timestamp...)
	  or default everything to STRING (safer/more stable against surprising type-inference
	  changes across batches, but requires you to cast explicitly downstream).

	cloudFiles.schemaEvolutionMode -- controls what happens when Auto Loader detects a NEW column
	  that wasn't in the previously-inferred schema:
		"addNewColumns"   (default) -- adds the new column to the schema, and FAILS the current
		                    stream run once so it can pick up the new schema on restart (this is
		                    intentional -- it wants you to consciously restart with the new schema
		                    rather than silently absorbing schema changes mid-stream)
		"rescue"          -- does NOT fail the stream; instead, puts any unrecognized/new column
		                    DATA into a special "_rescued_data" column (as a JSON string) so no
		                    data is ever silently DROPPED, while the stream keeps running uninterrupted
		"failOnNewColumns" -- stream fails and does NOT automatically update the schema at all --
		                    you must manually update the schema location before it can proceed
		"none"            -- ignores new columns entirely, doesn't update schema

	THE "_rescued_data" COLUMN -- shows up automatically whenever Auto Loader (or the PERMISSIVE
	read mode more broadly, Part B2) encounters data that doesn't cleanly fit the expected/known
	schema -- e.g. a field with an unexpected data type, or (in rescue mode above) a genuinely new
	column. Rather than silently dropping or nulling that data, it's preserved as a raw JSON blob
	in this column so you can inspect/reprocess it later -- a very practical, commonly-asked-about
	safety net for "how do you make sure Auto Loader never silently loses data on a schema surprise."

K7. Reading the Spark UI -- how to actually debug a slow job (a genuine practical gap -- Part C10
    covered explain() plans, but not the live monitoring UI itself, which is how most real
    debugging actually starts)

	The Spark UI (accessible per-cluster on classic compute, or via the Query Profile on
	serverless/SQL warehouses) has several tabs worth knowing by name and purpose:

		JOBS tab       -- one entry per ACTION you triggered (Part A7); shows how long each job
		                   took and how many stages/tasks it broke into -- your first stop to see
		                   WHICH action in your notebook is actually the slow one.
		STAGES tab     -- drill into a specific job's stages; shows shuffle read/write volume per
		                   stage -- a stage with a MUCH larger shuffle-write than expected is a
		                   strong signal something (a join, an aggregation) is more expensive than
		                   it should be.
		TASKS (within a stage) -- the real skew-detection tool: sort tasks by DURATION -- if most
		                   tasks finish in seconds but a handful take minutes, that's data SKEW
		                   (Part C8) or a STRAGGLER node (K3), visible directly as a duration
		                   outlier in this list.
		STORAGE tab    -- shows what's currently CACHED/PERSISTED (Part C5) and how much memory/
		                   disk it's consuming -- useful for spotting a stale, no-longer-needed
		                   cached DataFrame quietly eating executor memory.
		EXECUTORS tab  -- per-executor memory usage, GC time, and shuffle read/write -- an executor
		                   showing unusually high GC TIME relative to task time is a sign of memory
		                   pressure (too little memory for the workload, or too many small objects
		                   -- ties back to Kryo/serialization, K2, and Unified Memory Management, J3).
		SQL/DATAFRAME tab -- shows the actual query plan WITH live metrics overlaid on each
		                   operator (rows processed, time spent) -- the visual, "which operator in
		                   THIS specific query is the bottleneck" equivalent of explain() (Part C10).

	Interview-ready answer to "how do you debug a slow Spark job": start at JOBS to find the slow
	action, drill into its STAGES to find the expensive stage (usually a shuffle-heavy one), then
	look at that stage's TASKS sorted by duration to check for skew/stragglers, and check EXECUTORS
	for memory/GC pressure as a parallel hypothesis -- rather than guessing at config changes blind.

K8. Notebook workflows: %run vs dbutils.notebook.run() (a practical gap -- a very common way
    Databricks notebooks are actually chained together in real pipelines)

	%run ./other_notebook
		-- an INCLUDE-style mechanism: runs the OTHER notebook IN THE SAME SESSION/CONTEXT as the
		   calling notebook. Any variables, functions, or imports defined in the other notebook
		   become directly available in YOUR notebook afterward, as if you'd pasted its code in.
		-- must be its own cell, path is relative/absolute within the workspace, CANNOT be
		   parameterized with arguments and cannot be run conditionally inside other logic (it's
		   resolved essentially at "compile time" of the notebook, not truly dynamically).

	dbutils.notebook.run("./other_notebook", timeout_seconds=600, arguments={"param1": "value1"})
		-- runs the other notebook as a SEPARATE JOB/REMOTE CALL, in its OWN ISOLATED context
		   (does NOT share variables/session state with the caller) -- more like calling a
		   function than including a file.
		-- CAN pass arguments (read via widgets, Part B25, inside the called notebook) and CAN
		   capture a return value via that notebook's dbutils.notebook.exit(...) call.
		-- can be called CONDITIONALLY/dynamically (e.g. inside a Python if/loop), and runs on a
		   NEW ephemeral job cluster context by default in some configurations -- meaning it has
		   real orchestration/isolation overhead that %run doesn't.

	WHEN TO USE WHICH: %run for sharing common UTILITY code/helper functions across notebooks
	(like an "import" statement); dbutils.notebook.run() for actual ORCHESTRATION -- chaining
	genuinely separate notebook-based steps together, especially when you need isolation,
	parameters, or a return value, or need to call it conditionally.
	2026 note: for anything beyond simple ad hoc notebook chaining, Lakeflow Jobs (Part F13) with
	proper multi-task dependencies (K9) is the production-grade replacement for hand-rolling
	orchestration via dbutils.notebook.run() calls.

K9. Databricks Jobs / Lakeflow Jobs -- orchestration in depth (Part F13 only named the rename;
    here's the actual feature depth interviewers probe on)

	JOB CLUSTER vs ALL-PURPOSE CLUSTER (recap + depth, ties to Part A9) -- a job's tasks typically
	run on an EPHEMERAL job cluster spun up just for that run and torn down after (cheaper, no
	idle billing) rather than attaching to a shared, always-on all-purpose cluster.

	TASK TYPES a single job/pipeline can orchestrate: Notebook, Python script, JAR, SQL
	(query/dashboard/alert refresh), a Lakeflow Declarative Pipeline run, dbt, or ANOTHER job
	(nested job-as-a-task) -- a modern job is a DAG of heterogeneous task types, not just a
	single notebook.

	FOR-EACH TASK -- runs the SAME task definition repeatedly over a LIST of inputs (e.g. run the
	same ingestion notebook once per source system in a list), with configurable concurrency --
	the built-in answer to "how do I avoid copy-pasting the same task 20 times for 20 similar inputs."

	CONDITIONAL / IF-ELSE TASKS -- branch the DAG based on the outcome/output of a previous task,
	instead of every run always executing every task regardless of upstream results.

	TASK VALUES -- a lightweight way to pass small pieces of data BETWEEN tasks in the same job
	run (distinct from dbutils.notebook.run()'s return value, K8, which is for a single
	parent-child call) --
		dbutils.jobs.taskValues.set(key="row_count", value=12345)          # in task A
		count = dbutils.jobs.taskValues.get(taskKey="task_a", key="row_count")   # in task B

	RETRIES & NOTIFICATIONS -- configurable automatic retry count/backoff per task, and
	email/Slack/webhook notifications on start/success/failure -- standard production-
	orchestration features, worth naming even briefly since "how do you handle a flaky/failing
	step in a pipeline" is a common operational-maturity interview question.

	JOB PARAMETERS -- job-level parameters (distinct from per-notebook widgets, Part B25) that get
	passed down to whichever tasks declare they accept them, letting one job definition be
	reused across environments/contexts (e.g. a "run_date" parameter for a backfill run).

K10. Databricks-backed secret scopes (Part B6 only covered Azure Key Vault-backed scopes)

	Besides backing a secret scope with an EXTERNAL secret manager (Key Vault on Azure, or the
	equivalent on AWS/GCP), Databricks also supports a purely DATABRICKS-BACKED secret scope --
	secrets stored and encrypted directly within Databricks itself, no external Key Vault/Secrets
	Manager resource required:
		databricks secrets create-scope --scope my-scope
		databricks secrets put --scope my-scope --key my-key
	Retrieval inside a notebook is IDENTICAL either way: dbutils.secrets.get(scope, key) -- the
	calling code doesn't need to know or care whether the scope is Databricks-backed or
	externally-backed. Choose Key-Vault-backed when your org already centralizes ALL secrets
	there for cross-tool consistency/rotation policies; choose Databricks-backed for simplicity
	when Databricks is the only consumer.

K11. Service principals & OAuth (automation/CI-CD identity -- touched on briefly in Part J2/J21
     but not explained as its own concept)

	A SERVICE PRINCIPAL is a non-human identity used for AUTOMATED processes (a scheduled job, a
	CI/CD pipeline deploying via Databricks Asset Bundles, Part J21) instead of running things
	under a real person's individual user account -- so a job doesn't silently break when the
	human who originally created it leaves the company or changes their password, and so
	permissions/audit trails are scoped to "the deployment pipeline" rather than "Bob's account."
	Authentication for service principals (and increasingly for human users/CLI/API tooling too)
	is done via OAuth (short-lived, scoped tokens) rather than long-lived Personal Access Tokens
	(PATs) -- OAuth is now the RECOMMENDED approach over PATs for anything automated, since PATs
	are long-lived, broad-scoped secrets that are more dangerous if leaked.
	A JOB can be configured to "RUN AS" a service principal rather than the creating user's
	identity -- the standard production pattern, so a job's data access is governed by a stable,
	auditable, non-personal identity via Unity Catalog (Part E), not tied to any one employee.

K12. Databricks SQL materialized views (a real gap -- distinct from Lakeflow Declarative
     Pipelines' materialized views, Part F7, which only exist WITHIN a pipeline)

	Databricks SQL (DBSQL) also supports creating a materialized view DIRECTLY via SQL, OUTSIDE
	of any Lakeflow Declarative Pipeline -- for analysts/BI use cases that want a pre-computed,
	auto-refreshing result set without building a full pipeline:
		CREATE MATERIALIZED VIEW my_catalog.my_schema.mv_daily_sales AS
		SELECT order_date, SUM(amount) AS total_sales FROM orders GROUP BY order_date;
	Databricks handles incrementally refreshing it on a schedule (similarly to how a Lakeflow
	Declarative Pipeline materialized view stays current, Part F7) -- the key DIFFERENCE is this
	one is a standalone catalog object manageable straight from SQL/DBSQL, not something that
	only exists as part of a larger pipeline's dependency graph. Worth distinguishing clearly in
	an interview if asked "what's a materialized view on Databricks" -- there are genuinely two
	related-but-distinct implementations depending on context.

K13. Databricks Clean Rooms (a genuine, important gap -- not mentioned anywhere in Parts A-J)

	A CLEAN ROOM lets TWO OR MORE different organizations collaborate on and analyze SHARED data
	WITHOUT either party being able to see the other's raw, underlying data directly -- built on
	top of Delta Sharing/OpenSharing (Part E11) for the underlying data-movement mechanics, plus
	strict, jointly-agreed-upon computation controls.

	Classic use case: two companies (e.g. an advertiser and a publisher, or two healthcare
	organizations) want to compute a JOINED analysis across their respective customer/patient data
	(e.g. "how many of the advertiser's customers also appear in the publisher's audience") --
	WITHOUT either side ever handing over its raw customer list to the other. A clean room lets
	both parties contribute their data into a shared, governed, restricted compute environment
	where only APPROVED queries/notebooks (agreed to by both parties in advance) can run, and only
	AGGREGATE, non-identifying results ever leave the clean room -- neither party can run an
	arbitrary query to extract the other's raw rows.
	Interview-relevant framing: this is Databricks' answer to "privacy-preserving data
	collaboration between separate companies," a genuinely different problem from Unity Catalog's
	governance (which governs access WITHIN or across parts of the SAME organization's Databricks
	account) or plain Delta Sharing (which shares data but doesn't inherently restrict WHAT
	COMPUTATIONS a recipient can run against it once shared).

K14. Databricks Marketplace

	An open marketplace (built on the Delta Sharing/OpenSharing protocol, Parts E11) where
	providers can publish and organizations can DISCOVER/subscribe to shareable DATA (public/
	commercial datasets), AI MODELS, and Databricks-native APPLICATIONS -- directly consumable
	inside a workspace via the same Delta Sharing mechanics used for private inter-org sharing
	(Part E11), rather than through a separate, bespoke procurement/data-delivery process per
	vendor. Relevant mostly as awareness -- "Databricks has a marketplace for this too" -- rather
	than deep technical depth for most data-engineering interviews.

K15. Data classification & automated PII tagging in Unity Catalog

	Beyond MANUALLY defining ABAC policies/row filters/column masks (Parts E7/J24), Unity Catalog
	increasingly supports (semi-)AUTOMATED data classification -- scanning column names/content
	patterns/sample values to SUGGEST or auto-apply sensitivity TAGS (e.g. flagging a column that
	looks like an email address or SSN as "PII" automatically), which downstream governance
	policies (masking rules, access policies) can then key off of, rather than every team manually
	hunting for and tagging every sensitive column by hand across a large estate. Ties directly
	into the Business Glossary/Domains concepts (Part E10) -- classification tags are one more
	kind of metadata Genie Ontology (AI notepad Part F2) can also draw on for context-aware,
	privacy-respecting behavior.

K16. Unity Catalog admin hierarchy (a genuine gap -- Part E2 covered SETTING UP a metastore, but
     not the actual admin ROLE hierarchy on top of it)

		ACCOUNT ADMIN     -- highest level; manages the Databricks ACCOUNT itself (billing,
		                       workspaces, identity/SSO configuration, creating metastores).
		METASTORE ADMIN   -- manages a specific Unity Catalog METASTORE (Part E2) -- can create/
		                       manage catalogs within it, assign catalog-level ownership, and
		                       manage metastore-wide settings.
		CATALOG OWNER      -- owns a specific CATALOG (Part E3's catalog.schema.table namespace)
		                       -- can create schemas within it, grant access, manage its objects
		                       -- typically delegated to a specific team/domain (ties directly to
		                       the Data Mesh "domain ownership" principle, Part G4).
		WORKSPACE ADMIN    -- manages a specific WORKSPACE (compute policies, workspace-level
		                       settings, which users/groups can access that workspace) -- NARROWER
		                       in scope than a metastore admin, and workspace admin does NOT
		                       automatically grant Unity Catalog data permissions (data governance
		                       is deliberately SEPARATED from workspace administration -- a
		                       workspace admin can't automatically read every table just because
		                       they administer the workspace those queries run in).
	Interview-relevant point: this SEPARATION of workspace administration from data governance
	(you can admin the compute environment without automatically having data access, and vice
	versa) is a deliberate Unity Catalog design principle, and a good one to be able to explain
	if asked "how does Databricks separate infrastructure admin from data governance."

K17. Bloom filter indexes (legacy) + clarifying CACHE TABLE vs Delta Cache (two distinct things
     that get conflated)

	BLOOM FILTER INDEXES -- a LEGACY Delta Lake feature (predates Liquid Clustering, Part D21) --
	a probabilistic data structure attached to a column that can quickly tell you "this value is
	DEFINITELY NOT in this file" (with zero false negatives, but occasional false positives),
	letting Spark skip reading files that provably can't contain a value you're filtering for --
	most useful on high-cardinality columns used in point lookups (e.g. a UUID or email column)
	where Z-Ordering/Liquid Clustering's range-based co-location doesn't help as much. Largely
	superseded by Liquid Clustering for NEW tables in 2026, but still a real, valid, occasionally-
	asked-about feature for existing tables/legacy knowledge checks.

	CACHE TABLE vs DELTA CACHE (Part C11) -- genuinely different mechanisms, worth distinguishing
	explicitly since the names invite confusion:
		CACHE TABLE my_table;    -- plain SPARK-level in-memory caching (equivalent to df.cache(),
		                             Part C5) of a table's data for the CURRENT session -- gone
		                             when the session/cluster ends, exists in ANY Spark
		                             environment, not Databricks-specific.
		Delta Cache (Part C11)  -- Databricks-SPECIFIC, copies remote Delta/Parquet FILES onto
		                             worker nodes' LOCAL DISK (not JVM memory), persists across
		                             MULTIPLE queries/sessions as long as the same files are being
		                             read on the same cluster, and requires no explicit CACHE TABLE
		                             call at all once enabled -- it's a transparent file-level disk
		                             cache, not a query-result/DataFrame-level memory cache.
	One-liner: CACHE TABLE caches a RESULT in memory for your session; Delta Cache caches the
	underlying FILES on local disk, transparently, across sessions.

K18. GDPR / "right to be forgotten" pattern with Delta Lake

	A genuinely common REAL-WORLD requirement (and a good interview scenario question: "a user
	requests their data be deleted under GDPR -- walk me through how you'd actually guarantee
	that on a Delta Lake table with time travel enabled") that ties several earlier concepts
	together:
		1. run the actual DELETE (Part D7) -- this is a soft-delete at first (Part D2), so the
		   physical file with that user's data still exists on disk, and TIME TRAVEL (Part D11)
		   could still expose the "deleted" record by querying an older version.
		2. to TRULY, physically remove it, you need to bring the RETENTION WINDOW down and run
		   VACUUM (Part D14) with a short/zero retention (accepting that you lose time-travel
		   ability for versions before the deletion) -- this is the only way to guarantee the
		   underlying Parquet bytes are actually gone from storage, not just logically invalidated.
		3. an alternative/complementary technique for especially sensitive data: CRYPTO-SHREDDING
		   -- encrypt a given user's sensitive data with a PER-USER encryption key from the start,
		   and "delete" that user's data by simply DESTROYING their specific key -- the ciphertext
		   bytes can remain on disk forever (satisfying "we never rewrite/touch that Parquet file")
		   but become permanently, cryptographically unreadable the instant the key is destroyed --
		   a technique specifically suited to immutable/append-heavy storage where physically
		   rewriting every historical file for every deletion request would be impractical at scale.
	Interview-ready one-liner: "logical DELETE alone isn't enough for a compliance guarantee on
	Delta Lake, because time travel and the default retention window mean the data is still
	physically recoverable -- you need VACUUM with a short retention window, or crypto-shredding
	for data where you can't/won't ever rewrite the underlying files."

K19. Metadata-driven / parameterized pipelines (a common production PATTERN, not a single feature
     -- worth knowing as a design approach interviewers ask about directly: "how would you avoid
     writing 50 nearly-identical pipelines for 50 similar source tables?")

	Instead of hand-writing a separate notebook/pipeline PER source table (50 nearly-identical
	copies of the same bronze-to-silver logic, differing only in table/column names), store the
	VARYING details (source table name, target table name, key columns, watermark column, file
	format, etc.) as METADATA -- typically rows in a small control/config Delta table -- and write
	ONE generic, parameterized pipeline/notebook that LOOPS over that metadata table, using
	widgets (Part B25) or plain Python variables to drive each iteration's specific behavior:
		configs = spark.table("control.pipeline_config").collect()
		for cfg in configs:
		    run_generic_ingestion(source=cfg.source_table, target=cfg.target_table,
		                           key_cols=cfg.key_columns, watermark_col=cfg.watermark_column)
	Benefits: adding a 51st source table becomes "add a row to the config table," not "write and
	test a whole new pipeline"; a bug fix to the shared logic fixes ALL 50 pipelines at once
	instead of needing 50 separate code changes. Lakeflow Jobs' FOR-EACH task type (K9) is the
	modern, managed-orchestration way to implement exactly this pattern, rather than a hand-rolled
	Python for-loop inside one giant notebook.

K20. Pricing tiers & a brief note on networking (rounding out platform-level awareness)

	DATABRICKS PRICING TIERS (naming/exact features shift -- verify current tier boundaries
	against docs before relying on specifics in an interview, but the SHAPE is worth knowing):
		Standard    -- core platform capabilities.
		Premium     -- adds Unity Catalog's full RBAC/ABAC governance model, role-based access
		                 control at the workspace level, and other enterprise security features.
		Enterprise  -- adds the most advanced security/compliance controls (e.g. enhanced audit,
		                 compliance certifications for regulated industries).
	Billing itself is denominated in DBUs (Databricks Units) -- a normalized unit of processing
	capacity consumed per second, with the DBU RATE varying by compute type (serverless vs
	classic), instance type, and tier -- i.e. "how much you pay" = DBU rate x DBUs consumed x
	cloud infrastructure cost, not a single flat number.

	NETWORKING (brief, for platform/infra-flavored interview rounds specifically) -- for
	production/enterprise deployments, Databricks supports VPC/VNet INJECTION (deploying the
	Databricks-managed compute plane INTO your own organization's existing virtual network rather
	than a Databricks-owned one, for tighter network control) and PRIVATE LINK / Private Network
	Gateway (companion notepad Part I2) for keeping traffic between your workspace and cloud
	services off the public internet entirely. Deep networking design is usually a platform-team/
	cloud-infra interview topic rather than a data-engineer one, but knowing these terms exist
	signals broader platform fluency if it comes up.
