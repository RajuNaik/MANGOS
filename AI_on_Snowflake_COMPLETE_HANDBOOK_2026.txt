A1. AI/ML/GenAI, quick refresher (condensed -- full beginner explanation lives in the Databricks
    AI notepad Part A1; repeated here briefly so this file stands alone)

	AI (broadest) > Machine Learning (learns from data) > Deep Learning (neural networks) > GenAI/
	LLMs (generates new content). This whole handbook is mostly about the GenAI/LLM layer (Cortex
	AI, Parts B-C, E-F) plus Snowflake's classical-ML tooling (Snowflake ML, Part D) that predates
	and still underpins a lot of production AI work.
======================================================================================================================================================
A2. Why AI on Snowflake -- the platform's actual thesis (know this for "why Snowflake for AI" questions)

	Stated directly by CEO Sridhar Ramaswamy at Summit 2026: "The model is not your unique
	advantage. It's when you combine models with YOUR DATA that things begin to shine." Models
	themselves are increasingly commoditized (any vendor can call GPT/Claude/Gemini); the
	differentiator is GOVERNED ENTERPRISE DATA plus the CONTEXT and IDENTITY that make an agent's
	answers trustworthy.
	Practically, this means: AI functions/agents run INSIDE the same security/governance
	perimeter as your tables (Part B7) -- an agent's data access is governed by the exact same
	RBAC/masking/row-access policies as a human analyst's, automatically, with zero separate
	configuration. This is the direct Snowflake analog to the Databricks AI notepad's "why do AI
	work on a data platform at all" argument (Databricks AI notepad Part A2) -- both companies
	converged on the identical pitch: keep AI inside the governed platform instead of bolting on
	a separate stack that has to be kept in sync.
======================================================================================================================================================
A3. Cortex AI vs Snowflake ML -- the two broad categories (know this split, it structures this
    whole handbook and is a common opening interview question: "what AI/ML capabilities does
    Snowflake have?")

		CORTEX AI  -- Snowflake's managed, GenAI-native layer: LLM functions callable from SQL
		              (Part B), higher-level managed services (Cortex Analyst/Search/Document AI,
		              Part C), and the agentic surfaces (Cortex Agents, CoWork, CoCo -- Parts E-F).
		              You call a function or API; Snowflake manages the model/infrastructure.
		SNOWFLAKE ML -- the TRADITIONAL ML/MLOps layer (Part D): train your OWN classical ML models
		              (scikit-learn/XGBoost/PyTorch/etc.) using Snowpark ML, track them in the
		              Feature Store + Model Registry, deploy for inference. This existed BEFORE
		              Cortex AI and still underpins production ML work that isn't "call an LLM."
	Both layers run on GOVERNED data, inside the same RBAC/security perimeter, and increasingly
	interoperate (e.g. Agentic ML via CoCo, Part D, orchestrates Snowflake ML steps using natural
	language; Cortex Agents, Part C5, can call a Snowflake ML model as a tool).

PART B -- CORTEX AI FUNCTIONS (AISQL) -- the SQL-native AI layer
======================================================================================================================================================
======================================================================================================================================================
B1. AI_COMPLETE -- the general-purpose LLM call

	SELECT AI_COMPLETE('claude-4-5-sonnet', 'Summarize this: ' || review_text) FROM reviews;
	SELECT AI_COMPLETE('gemini-3-pro',
	  PROMPT('Summarize this issue shown in this screenshot in one sentence: {0}', img_file)
	) AS summary FROM images;
	Generates a completion for a text OR image input using a selected LLM -- the Snowflake analog
	to the Databricks AI notepad's ai_query() (Part B4). 2026 addition: AI_COMPLETE now also
	accepts AUDIO and VIDEO as inputs, not just text/images -- multimodal in the full sense
	(sentiment, classification, content insights on any of the four modalities, all as one SQL
	function call).
	Model choice: Cortex hosts models from Anthropic (Claude), OpenAI (GPT), Google (Gemini),
	Meta (Llama), Mistral AI, DeepSeek, xAI (Grok, GA 2026), and Snowflake's own open-source
	Arctic model -- all deployed WITHIN the Snowflake service perimeter (your data never leaves
	Snowflake to reach an external model API).
	STRUCTURED OUTPUT -- AI_COMPLETE supports a response-format/schema parameter to force
	schema-conformant JSON output, directly analogous to the Databricks AI notepad's "structured
	outputs" gap-filler (Databricks AI notepad Part I6) -- same problem (need RELIABLE, parseable
	output for a downstream pipeline column, not "hope the prompt was followed"), same solution
	shape, different function name.
	CORTEX GUARD -- AI_COMPLETE supports a guardrails option for content-safety filtering on
	output, the Snowflake analog to the Databricks AI notepad's built-in guardrails discussion
	(Databricks AI notepad Part E3).
======================================================================================================================================================
B2. AI_CLASSIFY, AI_FILTER, AI_AGG, AI_SENTIMENT -- task-specific functions

	AI_CLASSIFY(text_or_image, ARRAY('billing','technical','shipping'))   -- zero-shot classification
	AI_FILTER(text_or_image, 'Is this a complaint about a refund?')       -- returns TRUE/FALSE,
	                                                                          usable directly in a
	                                                                          WHERE/SELECT/JOIN...ON clause
	AI_AGG(text_column, 'Summarize the common themes across these reviews')  -- aggregates INSIGHTS
	                                                                          across MANY rows in one
	                                                                          call -- NOT subject to
	                                                                          context-window limits
	                                                                          (Snowflake chunks/
	                                                                          combines internally)
	AI_SENTIMENT(text)                                                    -- sentiment scoring
	GOTCHA worth knowing precisely: AI_FILTER being usable directly inside a WHERE clause is a
	genuinely distinctive Snowflake capability -- you can write
		SELECT * FROM support_tickets WHERE AI_FILTER('Is this urgent?', ticket_text);
	filtering rows with a plain-language predicate, the same way you'd write a normal WHERE clause
	-- there's no equally terse Databricks AI notepad equivalent (the closest is combining
	ai_classify() with a separate WHERE, not a single boolean-returning predicate function).
======================================================================================================================================================
B3. AI_EXTRACT, AI_TRANSLATE, AI_TRANSCRIBE

	AI_EXTRACT(text_or_document, ARRAY('vendor_name','invoice_amount','due_date'))
	  -- pulls structured fields out of unstructured text/documents based on plain-language
	     extraction instructions -- powered by Snowflake's proprietary ARCTIC-EXTRACT model,
	     which Snowflake reports outperforms general-purpose LLMs on document-understanding
	     benchmarks (e.g. DocVQA) because it understands VISUAL LAYOUT and context, not just raw
	     text -- direct analog to the Databricks AI notepad's ai_parse_document() (Databricks AI
	     notepad Part I9), just under a different function name and folded into the more general
	     AI_EXTRACT rather than a separate document-specific function.
	AI_TRANSLATE(text, 'es')          -- translate text between languages
	AI_TRANSCRIBE(audio_or_video_file) -- transcribes audio/video FROM A STAGE, extracting text,
	                                       timestamps, AND speaker information (diarization) --
	                                       GA'd alongside AI_CLASSIFY in November 2025.
======================================================================================================================================================
B4. AI_EMBED and similarity functions

	AI_EMBED(text_or_image)                          -- generates an embedding vector, usable for
	                                                     similarity search/clustering/classification
	AI_EMBEDDING_SIMILARITY / AI_SIMILARITY           -- cosine-similarity-style comparison between
	                                                     two embeddings
	This is the raw building block Cortex Search (Part C3) is built on top of -- same relationship
	as the Databricks AI notepad's Vector Search being built on embeddings (Databricks AI notepad
	Part A8/B5): most of the time you'll reach for the higher-level managed service (Cortex
	Search) rather than calling AI_EMBED yourself, but knowing the primitive underneath it is
	exactly what it is matters for interview depth.
======================================================================================================================================================
B5. Helper functions

	PROMPT('Summarize this: {0}', text_column)   -- builds a structured prompt OBJECT (rather than
	                                                 raw string concatenation) for use with
	                                                 AI_COMPLETE and similar functions -- safer/
	                                                 cleaner than manually concatenating strings,
	                                                 especially with multimodal inputs (images/files).
	TO_FILE(stage_path)                          -- creates a reference to a file in a stage
	                                                 (internal or external) for use with AI_COMPLETE
	                                                 and other file-accepting functions.
	AI_COUNT_TOKENS(text, model)                  -- returns token count for a given input against a
	                                                 specific model/function's limits, so you can
	                                                 pre-check/truncate before a call fails or gets
	                                                 unexpectedly expensive -- ties directly into
	                                                 cost management (B9).
======================================================================================================================================================
B6. FILE data type -- native multimodal column type

	A first-class SQL column type holding references to unstructured files (images, OCR-ready
	PDFs, audio/video blobs) directly in structured tables, natively usable as input to Cortex AI
	Functions -- e.g. SELECT AI_CLASSIFY(img_file, 'Scratch,Dent,Perfect') FROM inspection_photos.
	Direct analog to the Databricks AI notepad's FILE type discussion (Databricks AI notepad Part
	B1, tied to the Databricks platform notepad's FILE type, Part D27) -- both platforms
	independently added a native "unstructured data as a governed column type" primitive around
	the same period, for the same reason: let AI Functions/pipelines treat unstructured data as a
	normal, queryable, governed column rather than an external path reference.
======================================================================================================================================================
B7. Governance inheritance -- arguably Snowflake's single strongest AI differentiator, know it cold

	Cortex AI Functions inherit Snowflake's EXISTING security model automatically: role-based
	access control, column-level masking policies, row access policies, and network policies ALL
	apply to Cortex AI Functions with ZERO additional configuration. If a role doesn't have SELECT
	access to a column, that role CANNOT run an AI function against that column either -- period,
	no separate AI-specific access-control layer to build/maintain.
	GOTCHA/interview framing: this is functionally the SAME end goal as the Databricks AI
	notepad's Unity AI Gateway (Databricks AI notepad Part E3) -- "govern what an agent/function
	can see the same way you govern what a human can see" -- but the MECHANISM is different and
	worth stating precisely: Unity AI Gateway is a SEPARATE, purpose-built governance layer SITTING
	ON TOP of Unity Catalog specifically for AI/agent runtime behavior (spend caps, routing,
	contextual approval policies). Snowflake's approach is that Cortex Functions DIRECTLY REUSE the
	exact same RBAC/masking/row-policy engine that governs ordinary SQL queries -- there's no
	separate "AI governance layer" object model at the Cortex-Functions level; it's the SAME
	governance, automatically extended. (Snowflake's Agent Identity, Part E3, IS a more Gateway-
	like separate layer -- but it governs AGENTS specifically, not the plain SQL-callable AI
	Functions covered in this Part B, which just inherit ordinary table-level RBAC.)
======================================================================================================================================================
B8. Privileges

	Calling Cortex AI Functions requires: the account-level USE AI FUNCTIONS privilege, PLUS one
	of two database roles:
		SNOWFLAKE.CORTEX_USER        -- access to ALL Cortex AI features (broadest)
		SNOWFLAKE.AI_FUNCTIONS_USER  -- narrower, AI-Functions-specific access
	By default, CORTEX_USER is granted to the PUBLIC role (meaning every user gets it
	automatically) -- if you want to RESTRICT Cortex access to specific roles/teams, you must
	explicitly REVOKE it from PUBLIC first, then grant it selectively. A genuinely easy-to-miss
	governance gotcha: Cortex access is ON by default account-wide unless an admin deliberately locks it down.
======================================================================================================================================================
B9. Cost model & batch processing guidance

	Cortex AI Functions are billed PER TOKEN (input + output), separate from warehouse compute
	credits. Monitor via the same account-level usage views as everything else (Snowflake platform
	notepad Part I2) plus dedicated Cortex/AI cost dashboards.
	Cortex AI Functions are explicitly OPTIMIZED FOR THROUGHPUT/BATCH use -- Snowflake's own
	guidance is to use these functions to process NUMEROUS inputs at once (e.g. a whole SQL table
	of text) rather than one-off single calls, since batch processing is architecturally what
	they're built for. This is the direct Snowflake analog to the Databricks AI notepad's "batch
	LLM inference at scale" operational guidance (Databricks AI notepad Part I7) -- same
	underlying advice (design for volume, not one-off calls), delivered as Snowflake's own
	explicit product guidance rather than something you have to work out yourself.
	Practical cost-control tips: monitor token usage via AI_COUNT_TOKENS (B5) before expensive
	calls on long documents; prefer task-specific functions (AI_CLASSIFY/AI_SENTIMENT/etc.) over
	the more general AI_COMPLETE where one fits, since task-specific functions are specifically
	optimized/cost-effective for their narrow job; automate via Tasks (Snowflake platform notepad
	Part D9) to run AI enrichment on a schedule against only NEW/CHANGED rows (via a Stream)
	rather than reprocessing a whole table repeatedly.

PART C -- CORTEX ANALYST, SEARCH, DOCUMENT AI & AGENTS -- the managed higher-level services
======================================================================================================================================================
======================================================================================================================================================
C1. Cortex Analyst -- text-to-SQL, the Snowflake analog to the Databricks AI notepad's Genie
    NL-to-SQL roots (Databricks AI notepad Part F1)

	A fully-managed, agentic service that converts a business user's natural-language question
	into accurate SQL and returns results -- built specifically to avoid the "generic AI solution
	struggling with text-to-SQL because it only has a raw schema, no business context" problem.
	The key mechanism: a SEMANTIC MODEL (C2) -- NOT just the raw database schema -- bridges the
	gap between business language and the actual tables/columns.
	GOVERNANCE/PRIVACY commitments worth stating precisely in an interview: Cortex Analyst does
	NOT train on customer data; at inference time it uses ONLY the semantic model's metadata
	(table/column names, types, descriptions) to generate the SQL query, which is then EXECUTED in
	YOUR OWN virtual warehouse -- the LLM never sees your actual row-level data, only the metadata
	describing its shape.
	If configured across MANY data sources/semantic models, Cortex Analyst can automatically
	figure out WHICH one a given question should route to -- you don't have to specify the right
	source per query.
	Privileges: SNOWFLAKE.CORTEX_USER (broad) or SNOWFLAKE.CORTEX_ANALYST_USER (narrower, Analyst-only).
	Realistic framing (worth having ready, straight from practitioner guidance): treat Cortex
	Analyst as handling ~70% of routine, well-modeled questions well; complex ad hoc analysis still
	benefits from a skilled analyst writing SQL by hand -- and result quality depends ENTIRELY on
	how well-built the underlying semantic model (C2) and data foundation are, not on the tool
	being "magic."
======================================================================================================================================================
C2. Semantic Models -- the YAML layer that makes Cortex Analyst actually work (a senior-DE-owned artifact)

	A lightweight YAML file describing tables/columns similarly to a database schema, but enriched
	with the business meaning a raw schema lacks: business term SYNONYMS, METRIC definitions
	(pre-built formulas, not just raw columns), RELATIONSHIPS between tables, and descriptions.
	WHY A DATA ENGINEER OWNS THIS: this is the direct Snowflake analog to the Databricks AI
	notepad's Genie Ontology / Unity Catalog Metrics + Business Glossary combination (Databricks
	AI notepad Part F2, Databricks platform notepad Part E9-E10) -- a governed, explicit
	definition of what "revenue" or "active customer" MEANS at your company, so an AI agent
	doesn't have to guess or hallucinate business logic. Building and maintaining a good semantic
	model is squarely data-engineering work, the same way maintaining Unity Catalog Metrics
	definitions is on the Databricks side.
	2026 additions: SEMANTIC STUDIO (build shared business logic/semantic views without needing
	SQL expertise -- a lower-code authoring surface over the same underlying YAML) and SEMANTIC
	VIEW AUTOPILOT (automatically refines/improves semantic views over time based on usage
	patterns -- a Snowflake-specific twist on "keep the context layer accurate without 100%
	manual upkeep," conceptually adjacent to how Genie Ontology self-improves on the Databricks
	side, Databricks AI notepad Part F2).
	OSI (Open Semantic Interchange) -- a 2026-announced OPEN STANDARD (54 vendors backing it) for
	semantic-layer interoperability -- the industry-wide signal that semantic/context layers need
	to be portable across tools, not locked into one vendor's proprietary format.
======================================================================================================================================================
C3. Cortex Search -- hybrid retrieval, the RAG building block

	A managed search service indexing your Snowflake data for BOTH semantic (vector/embedding)
	similarity AND traditional keyword (BM25-style lexical) matching in ONE hybrid query -- direct
	analog to the Databricks AI notepad's Vector Search (Databricks AI notepad Part B5), with one
	notable difference worth flagging: Cortex Search is hybrid (vector + keyword) BY DEFAULT/
	built-in, whereas the Databricks AI notepad describes hybrid search as an additional
	capability layered onto Vector Search (Databricks AI notepad Part I5) rather than the
	out-of-the-box default behavior.
	Auto-indexed/synced against the underlying Snowflake table -- same "no manual ETL to keep
	embeddings fresh" value proposition as Databricks' Delta-Sync vector index.
	Standard RAG pattern on Snowflake: Cortex Search retrieves relevant chunks -> AI_COMPLETE (B1)
	generates a grounded answer using those chunks -> all governed by the same RBAC the source
	table already has (B7).
======================================================================================================================================================
C4. Document AI

	A managed service specifically for extracting structured data from DOCUMENTS (invoices,
	contracts, forms, reports) -- built on the same Arctic-Extract model underneath AI_EXTRACT
	(B3), but packaged as a purpose-built, higher-level workflow: define the fields you want
	extracted, point it at a document type, and it handles layout-aware extraction across many
	similar documents, with confidence scoring so low-confidence extractions can be routed to a
	human reviewer (a common production pattern: trigger an alert/notification for low-confidence
	extractions rather than silently trusting every automated result).
	Practical pipeline shape (a real senior-DE-relevant pattern worth knowing end-to-end): event-
	driven document landing (e.g. new file in an external stage) -> Document AI / AI_EXTRACT
	parses/classifies/extracts structured fields -> results land in a normal governed table ->
	Cortex Search indexes the documents for RAG/chat -> Streamlit (platform notepad Part F6)
	or a BI tool surfaces it. Runs as SERVERLESS Tasks (platform notepad Part D9) that auto-scale
	to handle variable document volume, with cost tracked per credit in the standard usage views
	(platform notepad Part I2).
======================================================================================================================================================
C5. Cortex Agents -- the orchestration layer underneath CoWork/CoCo

	An API that ORCHESTRATES Cortex Analyst (C1) + Cortex Search (C3) + custom tools into ONE
	agentic interface -- given a user's question, a Cortex Agent decides whether it needs
	structured data (route to Analyst), unstructured search (route to Search), or both, executes
	the necessary steps, and synthesizes a final answer. This is the BUILDING BLOCK that Snowflake's
	own flagship agent products (CoWork and CoCo, Part F) are built on top of -- direct analog to
	how the Databricks AI notepad's Mosaic AI Agent Framework (Databricks AI notepad Part D3)
	underlies Genie/Agent Bricks. If you need a CUSTOM agent (not CoWork/CoCo's own UI), Cortex
	Agents is the code-first API layer you'd build against directly.

PART D -- SNOWFLAKE ML (traditional ML / MLOps) -- predates Cortex AI, still core production tooling
======================================================================================================================================================
======================================================================================================================================================
D1. Snowpark ML -- the Python library underneath everything in this Part

	The snowflake-ml-python package. Two main pieces:
		- MODELING API (snowflake.ml.modeling) -- scalable wrappers around familiar frameworks
		  (scikit-learn, XGBoost, LightGBM) for data preprocessing/feature engineering/model
		  training, executed using Snowflake's OWN compute (Snowpark-optimized warehouses, D7) --
		  same "push compute to the data, no data movement" philosophy as Snowpark DataFrames
		  (platform notepad Part F3).
		- FRAMEWORK CONNECTORS -- optimized, secure data provisioning for PyTorch/TensorFlow in
		  their NATIVE data-loader formats, for deep-learning workloads that need a framework
		  Snowpark ML's own Modeling API doesn't wrap directly.
	Median reported gains (Snowflake's own benchmarking): ~3.5x faster performance and ~34% lower
	cost for AI/ML and data engineering workloads vs. moving data out to a separate ML environment
	-- the same "process where the data lives" argument the Databricks AI notepad makes for
	Feature Store pipelines (Databricks AI notepad Part I2).
======================================================================================================================================================
D2. Feature Store -- Entities, Feature Views, point-in-time correctness

	Requires Enterprise Edition+ (Snowflake platform notepad Part A5).
	ENTITY -- the "thing" features are about (e.g. customer, product) -- defines the join key.
	FEATURE VIEW -- a versioned, managed definition of one or more features tied to an Entity,
	backed by a Snowpark DataFrame transformation -- Snowflake continuously/automatically refreshes
	it (batch OR streaming) so the feature stays current.
	POINT-IN-TIME CORRECT training set generation -- the exact same problem/solution shape as the
	Databricks AI notepad's FeatureLookup mechanism (Databricks AI notepad Part I2): you generate
	a training dataset by joining labels against Feature Views AS OF each label's own timestamp,
	avoiding training-serving skew / label leakage, without hand-writing the correlated-timestamp
	join yourself.
	ONLINE SERVING (2026: GA, sub-2-second freshness, ~10ms serving latency) -- the real-time
	inference-time counterpart, direct analog to the Databricks AI notepad's Feature Serving
	(Databricks AI notepad Part I3) -- both platforms converged on "offline Feature Store for
	training + a separate low-latency online store for real-time lookups" as the standard shape.
	CLUSTER_BY support on Feature Views (a 2026 addition) lets you apply the same clustering-key
	performance thinking (platform notepad Part C1) to feature data specifically.
======================================================================================================================================================
D3. Model Registry -- versioned model management, works for models trained ANYWHERE

	from snowflake.ml.registry import Registry
	reg = Registry(session=session)
	mv = reg.log_model(my_sklearn_model, model_name="churn_model", version_name="v1",
	                    metrics={"accuracy": 0.91})
	Logs and manages ML models regardless of whether they were trained INSIDE Snowflake (Snowpark
	ML) or externally (any model you bring in) -- direct analog to the Databricks AI notepad's
	MLflow Model Registry (Databricks AI notepad Part A4/I4).
	VERSIONING + ALIASES -- each logged model gets versions; you can assign an ALIAS (a mutable,
	custom-named pointer, e.g. "default"/"champion") to a specific version:
		ALTER MODEL churn_model SET DEFAULT_VERSION = 'v1';         -- SQL
		# or via Python: mv.model.default = "v1"
	GOTCHA/direct comparison worth stating precisely: this is the SAME "aliases, not fixed stages"
	pattern the Databricks AI notepad specifically flags as a common outdated-tutorial trap
	(Databricks AI notepad Part I4, correcting the old Staging/Production/Archived vocabulary) --
	Snowflake's Model Registry was ALWAYS alias/version-based, never went through an equivalent
	"fixed stage enum" legacy phase the way MLflow's original Model Registry did. Worth knowing
	this history if asked to compare the two registries' design.
	INFERENCE -- call a registered model's method directly in SQL or Python:
		SELECT churn_model!predict(*) FROM customer_features;                    -- SQL
		remote_prediction = mv.run(test_features, function_name="predict")       -- Python, lazy-evaluated
	Runs INSIDE the warehouse specified by your session (for CPU-scale batch/warehouse inference)
	or can be deployed to SPCS (platform notepad Part F4) for real-time, potentially GPU-backed
	serving via a REST endpoint.
	SHARING -- models can be shared/replicated with granular privileges: USAGE (run inference via
	warehouse, no visibility into internals) vs READ (also usable for SPCS inference, can see metadata).
======================================================================================================================================================
D4. ML Observability, Explainability, and Lineage

	ML OBSERVABILITY -- monitor a deployed model's PERFORMANCE and DRIFT metrics in production,
	set alerts on performance thresholds -- direct analog to the Databricks AI notepad's
	production-monitoring discussion (Databricks AI notepad Part E5), applied to classical ML
	models specifically here rather than GenAI/agent traces.
	ML EXPLAINABILITY -- computes SHAPLEY VALUES for any model in the Model Registry, regardless
	of where it was originally trained -- the "why did the model predict this" answer for
	classical ML, distinct from the Databricks AI notepad's explicit note that explainability is
	structurally HARDER for LLMs than classical models (Databricks AI notepad Part I18) -- Shapley
	values are a classical-ML-specific tool that doesn't have a clean GenAI equivalent on either platform.
	ML LINEAGE -- traces end-to-end lineage from SOURCE DATA -> FEATURES -> DATASETS -> MODELS,
	enabling reproducibility/compliance/debugging across the full ML lifecycle -- direct analog to
	Unity Catalog's automatic lineage extended to ML artifacts (Databricks platform notepad Part E5).
======================================================================================================================================================
D5. Datasets -- versioned, immutable training snapshots

	Create a versioned, IMMUTABLE snapshot of a training dataset (e.g. as Parquet files) so a
	model's training run can always be exactly REPRODUCED later, even if the underlying source
	tables have since changed -- the Snowflake-native version of "pin your training data" hygiene,
	complementary to (but distinct from) Time Travel (platform notepad Part D2), which is
	about the SOURCE table's history, not a purpose-built, named, ML-specific training artifact.
======================================================================================================================================================
D6. ML Functions (pre-built, SQL-callable -- predate Cortex AI, still heavily used for classic tasks)

	SQL-callable functions for COMMON predictive tasks WITHOUT writing/training a custom model
	yourself:
		FORECASTING          -- time-series forecasting (e.g. predict next 30 days of sales)
		ANOMALY DETECTION    -- flag outliers/unusual patterns in a metric over time
		CLASSIFICATION       -- general-purpose classification against a labeled training set
		CONTRIBUTION EXPLORER -- explains WHICH dimensions/segments are driving a change in a
		                          metric (e.g. "why did revenue drop last week" broken down by
		                          region/product/channel)
	These are genuinely "no-code ML" -- callable directly as SQL functions against a table, no
	Snowpark/Python ML pipeline required. A real, practical differentiator worth knowing: for the
	common, well-understood task shapes (forecast a metric, detect anomalies, explain a
	contribution), you often don't need Snowpark ML/Feature Store/Model Registry at all -- just an
	ML Function call. Reach for the fuller Snowpark ML stack (D1-D5) when the task is more custom
	or needs a bespoke model architecture.
======================================================================================================================================================
D7. Training & inference compute options

	Standard virtual warehouses (platform notepad Part B) -- fine for lighter feature engineering/
	CPU-bound training.
	Snowpark-optimized warehouses -- HIGH-MEMORY warehouse variant specifically for memory-
	intensive ML training/preprocessing (large in-memory joins, big feature matrices) that would
	spill/struggle on a standard warehouse.
	Container Runtime on Snowpark Container Services (SPCS, platform notepad Part F4) -- for GPU-
	backed training (deep learning, large models) or real-time low-latency inference endpoints --
	the Snowflake analog to the Databricks AI notepad's serverless GPU compute (Databricks AI
	notepad Part C4) for fine-tuning/deep learning workloads.
======================================================================================================================================================
D8. Cortex Training (2026) -- managed fine-tuning of open models

	Fine-tune OPEN models (e.g. Qwen, Mistral) on Snowflake-MANAGED GPUs, directly inside the
	platform -- you don't provision/manage GPU infrastructure yourself. Direct analog to the
	Databricks AI notepad's fine-tuning discussion + serverless GPUs (Databricks AI notepad Part
	C4) -- same "request the fine-tuning job, the platform handles the GPU compute" pitch.

F3. Cortex Sense (private preview, entering broader availability) -- the runtime context layer
    underneath BOTH CoWork and CoCo -- arguably THE single most important 2026 AI concept to know deeply

	THE PROBLEM IT SOLVES: an AI agent can know your database SCHEMA but not the BUSINESS MEANING
	behind it -- e.g. a query for "Q3 revenue" fails or answers wrong if the agent doesn't know
	your company's fiscal calendar starts in February, or what your specific "revenue" metric
	formula actually is. Internal Snowflake benchmarking (cross-validated independently by
	Anthropic) found agents WITHOUT any context layer scored only ~21-25% accuracy on hard,
	cross-table, metric-formula-dependent business questions.
	WHAT IT DOES: automatically builds a SHARED CONTEXT SUBSTRATE from signals your business
	ALREADY produces -- query history (what your analysts have actually run before), metadata
	from your transformation tools, metrics already defined in your BI tools, and Horizon Context
	semantic views (C2) -- WITHOUT you having to hand-author a second system of record. Where a
	governed semantic view EXISTS, Cortex Sense treats it as the AUTHORITATIVE signal (semantic
	views remain the gold standard for governed, consistent answers); across the REST of your data
	estate (e.g. a table for a feature launched two weeks ago with no semantic view built yet
	yet), Cortex Sense builds understanding on its own automatically.
	SELF-CORRECTING LOOP: continuously identifies gaps/conflicts in its own knowledge, improving
	via three feedback sources -- your own gold-standard benchmarks, user feedback, and its own
	suggestions for thin-coverage areas.
	MEASURED IMPACT: raises agent accuracy from the ~21-25% ungrounded baseline to 83-86% on hard
	benchmark questions (numbers vary slightly by source/methodology, consistently in this range)
	-- and REDUCES COST at the same time (Snowflake-reported: ~$1.76 -> ~$0.59 per query vs. a
	frontier agent manually inspecting tables), because a grounded agent stops wastefully running
	DESCRIBE TABLE against dozens of objects trying to reverse-engineer what they mean. Upfront
	INDEXING has a one-time cost that Snowflake says typically pays for itself via lower per-query
	cost over time.
	GOTCHA/direct comparison worth having exactly right: this is THE technical answer behind the
	SAME "AI doesn't have an intelligence problem, it has a context problem" framing the Databricks
	AI notepad attributes to Genie Ontology (Databricks AI notepad Part F2) -- both Snowflake and
	Databricks independently built a "continuously-learned, automatically-assembled business
	context layer feeding an agent" capability and used NEAR-IDENTICAL keynote language to
	describe the underlying problem, in the SAME 2026 conference season. A genuinely striking
	convergence, worth naming explicitly if asked to compare the two platforms' AI strategies --
	this is probably the single best "these platforms are converging" talking point in either handbook.
	RELATIONSHIP TO HORIZON CONTEXT (F4) -- know this distinction precisely, it's a common
	follow-up question: Horizon Context is the GOVERNANCE/METADATA substrate (manages semantic
	views, metadata connectors, business glossary -- the durable, curated layer). Cortex Sense is
	the RUNTIME ENRICHMENT layer that ACTIVATES those signals FOR AGENTS AT QUERY TIME (plus
	everything it learns automatically beyond what's been formally curated). Cortex Sense is NOT a
	standalone product -- CoWork/CoCo are the user-facing INTERFACE; Cortex Sense is the context
	ENGINE running underneath both.

	HORIZON CONTEXT -- specifically the context/metadata layer ensuring ONE version of the truth

G1. The governance stack, top to bottom (a good structure to recite if asked "how is AI governed on Snowflake")

		Cortex AI Functions (Part B)   -- inherit ordinary table-level RBAC/masking/row-access
		                                   policies automatically (B7) -- no separate config.
		Cortex Analyst/Search (Part C) -- governed by the SAME RBAC on underlying tables; the LLM
		                                   only ever sees semantic-model METADATA, never raw
		                                   customer data, for query generation (C1).
		Agents (CoWork/CoCo/Cortex Agents, Parts E-F) -- governed by AI Agent Identity (E3,
		                                   cryptographic per-agent identity + audit) + the 5-layer
		                                   AI Security stack (E4) + MCP governance via Natoma (E2).
		Models (Snowflake ML, Part D)  -- governed via Model Registry privileges (USAGE/READ, D3)
		                                   + ML Observability/Lineage (D4).
	The throughline: Snowflake's philosophy is REUSE existing platform governance wherever
	possible (Cortex Functions inheriting RBAC directly, B7) and add PURPOSE-BUILT new governance
	only where genuinely new risk surfaces exist (agents taking autonomous actions -- Agent
	Identity + the 5-layer AI Security stack).
======================================================================================================================================================
G2. Model lifecycle & Behavior Change Releases (BCRs) -- an operational-maturity topic worth knowing

	Models powering Cortex features follow a defined LIFECYCLE: PREVIEW (for evaluation, may
	change frequently, not yet stable) -> GA (stable, suitable for production).
	Snowflake continuously updates the underlying models. A change counts as a BEHAVIOR CHANGE if
	it could affect required syntax (e.g. how you specify a model/version) or otherwise impact how
	you interact with a feature. These are communicated via formal BEHAVIOR CHANGE RELEASES (BCRs)
	specifically so customers can review/prepare BEFORE a change lands in their account, rather
	than discovering a breaking change in production. Worth knowing this exists as a genuine
	operational-maturity/production-readiness talking point -- "how do you keep a Cortex-dependent
	pipeline stable as Snowflake updates its underlying models" has a real, named answer (subscribe
	to and review BCRs), not just "hope nothing changes."
======================================================================================================================================================
G3. Data privacy commitments (know this precisely, it's a real trust/compliance talking point)

	Snowflake's stated commitment: Cortex AI features do NOT train on Customer Data -- your data
	is never used to train or fine-tune any model made available across Snowflake's broader
	customer base. All models run WITHIN Snowflake's security/governance perimeter (B1) -- data
	doesn't leave Snowflake to reach an external model provider's own infrastructure for standard
	Cortex usage. Direct analog to the Databricks AI notepad's data-privacy framing for Foundation
	Model APIs (Databricks AI notepad Part C1-C2) -- both platforms make essentially the same
	"your data doesn't train the shared model, and stays inside our perimeter" commitment.
======================================================================================================================================================
G4. Cost governance for AI specifically

	Token-based billing (B9) is TRACKED separately from warehouse credits but visible through the
	SAME account-level usage views (platform notepad Part I2) plus Cortex-specific cost
	dashboards. Resource Monitors (platform notepad Part B5) and Trust Center's posture monitoring
	(platform notepad Part G9) extend naturally to AI spend/risk oversight -- there's no separate
	"AI billing system" to learn; it's the same cost-governance toolkit applied to a new
	consumption category (tokens, alongside credits).

H3. Interview quick hits (Snowflake AI side)

	Q: "How does Snowflake govern what an AI agent can access?"
	A: Cortex AI Functions automatically inherit the SAME RBAC/masking/row-access policies as any
	   SQL query (B7) -- zero separate config. Agents specifically (which can take ACTIONS, not
	   just read) get AI Agent Identity (E3) -- cryptographic per-agent identity, per-agent RBAC,
	   full audit trail -- plus the 5-layer AI Security stack (E4).

	Q: "What's the difference between Cortex Sense and Horizon Context?"
	A: Horizon Context is the durable, CURATED governance/metadata substrate (semantic views,
	   glossary, connectors). Cortex Sense is the RUNTIME layer that activates those signals for
	   agents at query time, plus everything it learns automatically beyond what's been formally
	   curated (F3).

	Q: "CoWork vs CoCo -- what's the split?"
	A: CoWork = business-user personal agent (natural language, no SQL needed, Deep Research,
	   Artifacts). CoCo = developer/coding agent (generates/deploys code, Cloud Agents sandboxes,
	   Agent Teams for parallel multi-agent execution). Same underlying Cortex Agents API (C5) and
	   Cortex Sense context layer (F3) power both.

	Q: "How would you build a RAG chatbot over internal documents on Snowflake?"
	A: land documents governed via a stage/table -> Document AI or AI_EXTRACT for structured
	   fields (C4/B3) -> Cortex Search index for retrieval (C3) -> AI_COMPLETE to generate a
	   grounded answer using retrieved chunks (B1) -> Streamlit or CoWork/Cortex Agents (C5) as
	   the front end -- all governed by the SAME RBAC as the source documents throughout.

	Q: "Snowflake vs Databricks -- how do their AI governance strategies compare?"
	A: both landed on a two-surface split (one business-user agent, one developer agent) and a
	   context/semantic layer feeding both, within months of each other in 2026 -- Databricks via
	   Genie + Genie Ontology + Unity AI Gateway; Snowflake via CoWork/CoCo + Cortex Sense + Agent
	   Identity. Different implementations, remarkably similar underlying thesis: models are
	   commoditized, governed enterprise context is the moat (see platform notepad Part L3 for the
	   fuller platform-wide comparison).

	Q: "What's Snowflake's data privacy commitment for Cortex AI?"
	A: no training on customer data, ever, for models shared across the customer base; all
	   inference runs within Snowflake's own security perimeter (G3).
